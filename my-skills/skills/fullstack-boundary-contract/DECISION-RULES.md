# 功能归属决策规则

## 决策优先级

规则按优先级从高到低排列。高优先级规则命中即可判定，无需继续向下匹配。

---

## 优先级 1：安全强制规则（必须后端）

以下功能**无论任何情况**都必须在后端实现，前端不得承担核心逻辑：

| 功能类型 | 示例 | 原因 |
|----------|------|------|
| 密码/密钥处理 | 密码哈希、密钥签名、Token 签发 | 客户端代码可被逆向 |
| 权限/鉴权校验 | 接口访问控制、角色判断、资源所有权验证 | 前端校验可被绕过 |
| 金额/积分计算 | 订单金额、优惠计算、余额扣减 | 防止客户端篡改 |
| 防刷/限流 | 短信验证码频率、接口调用限制 | 客户端限制无效 |
| 敏感数据聚合 | 跨用户数据查询、数据脱敏规则 | 数据隔离要求 |
| 第三方服务调用 | 支付、短信、邮件发送 | 密钥不可暴露 |

---

## 优先级 2：数据归属规则

**数据需要持久化 → 后端**

- 写入数据库、文件系统、缓存（Redis 等服务端缓存）
- 跨会话保持的状态（用户设置、历史记录）
- 多端同步的数据

**数据仅在当前会话/页面使用 → 前端**

- 表单草稿（未提交前）
- UI 展开/折叠状态
- 当前页面的筛选条件（不需要分享/书签）
- 客户端缓存（localStorage、sessionStorage、内存）

---

## 优先级 3：计算复杂度规则

**业务规则计算 → 后端**

- 涉及数据库数据的计算（统计、聚合、排名）
- 需要保证一致性的计算（库存扣减、并发控制）
- 计算结果需要被其他服务消费

**展示层计算 → 前端**

- 纯格式化（日期格式、数字千分位、货币符号）
- 基于已有数据的派生展示（百分比、进度条宽度）
- 排序/筛选（数据量小、无需服务端分页时）

---

## 优先级 4：交互性质规则

**纯 UI 交互 → 前端**

- 表单字段格式校验（手机号格式、邮箱格式、必填项）
- 页面路由和导航
- 动画、过渡效果
- 骨架屏、Loading 状态、空状态展示
- 无障碍属性（aria-label 等）
- 响应式布局

**业务规则校验 → 后端**（前端可做 UI 层预校验，但后端必须兜底）

- 用户名唯一性校验
- 库存是否充足
- 优惠券是否可用
- 业务状态流转合法性

---

## 优先级 5：协作模式判断

当功能同时涉及前后端时，按以下模式拆分：

### 模式 A：前端触发 + 后端执行
适用：文件上传、表单提交、操作确认

```
前端职责：UI 交互、进度展示、错误提示
后端职责：数据处理、持久化、返回结果
接口：POST /api/resource
```

### 模式 B：后端数据 + 前端渲染
适用：列表展示、详情页、数据看板

```
前端职责：渲染、分页 UI、筛选 UI、空状态
后端职责：数据查询、分页逻辑、权限过滤
接口：GET /api/resource?page=1&size=20
```

### 模式 C：实时双向通信
适用：消息推送、实时协作、进度通知

```
前端职责：WebSocket 连接管理、消息渲染、重连逻辑
后端职责：事件推送、连接管理、消息持久化
接口：WS /ws/channel/{id}
```

### 模式 D：前端状态 + 后端同步
适用：购物车、草稿、用户偏好设置

```
前端职责：本地状态管理、乐观更新、冲突提示
后端职责：持久化、冲突解决、多端同步
接口：PUT /api/resource/{id}（带版本号/时间戳）
```

## 特殊场景：SSR / BFF 层

若项目有 SSR（服务端渲染）或 BFF（Backend for Frontend）层，需额外说明：

- **SSR 页面**：首屏数据获取在服务端，但交互逻辑仍在客户端
- **BFF 层**：BFF 承担接口聚合、数据裁剪，不承担核心业务逻辑
- **边界原则**：BFF 属于"前端侧后端"，核心业务规则仍应在领域服务（Domain Service）中

## 灰色地带处理

遇到难以判断的功能点，使用以下问题链：

1. **"如果前端被绕过，会有安全/数据问题吗？"** → 是 → 后端
2. **"这个逻辑需要访问数据库或其他服务吗？"** → 是 → 后端
3. **"这个逻辑在多个客户端（Web/App/小程序）都需要吗？"** → 是 → 后端
4. **"这个逻辑只影响当前用户的当前页面体验吗？"** → 是 → 前端
5. 以上都不确定 → 标记为**协作**，在契约中明确各自边界
