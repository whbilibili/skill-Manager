| G10 | diff 删除了状态清理/重置行 | diff 中 `-` 掉了 `this.xxx = undefined/null/[]`、`reset()`、`clear()`、`dispose()`、`runInAction(...)` 内的清零语句，且没有在同一个函数里用等价语义替换 |

## 第 0 步：生成 diff + 收集改动意图 + 判断审查模式 + 初始化进度文件

### 0.0 范围识别（审查对象限定，防止全仓扫描浪费 context）

从用户当前消息 / PR 描述 / commit message 里抽取**范围信号**，决定本次 CR 只扫哪些路径：

| 信号类型 | 样例 | 处理方式 |
|---------|------|---------|
| **路径直指** | "只看 `packages/container/pc/src/pages/profit/`"、"focus on `src/stores/`" | 转为 git pathspec，赋值 `SCOPE="packages/container/pc/src/pages/profit/ src/stores/"` |
| **模块关键词** | "只审 profit 模块"、"只看广告账户相关的改动" | 用 `git diff --name-only master...HEAD` 拿改动文件，用关键词 `grep` 匹配得到路径集合后赋 `SCOPE` |
| **功能前缀** | "只看 UI 层"、"只看 Store 层"、"只看 api 层" | 映射到团队约定路径：UI → `packages/container/`；Store → `packages/share/module/vm/`；API → `packages/share/api/` |
| **通用信号**（无具体路径）| "范围里别扫 test"、"忽略 generated" | 加到 `EXCLUDE` 排除列表 |
| **未出现任何范围信号** | 常见情况 | `SCOPE=""`（全量），但必须在第 0 步末尾**主动问用户一句**："本次 CR 是否需要限定在某个目录 / 模块？全量扫描对于大 PR 噪声会高。若不限定回 `全量` 即可。" |

**范围声明块**（0.0 产物，必须输出到报告开头）：
```
━━━ 审查范围 ━━━
scope:   packages/container/pc/src/pages/profit/   （用户指定 / AI 映射自 "profit 模块" / 全量）
exclude: __mocks__/**  *.stories.tsx               （或 "无"）
━━━━━━━━━━━━━━
```

### 0.1 生成 diff（用 git CLI，避开 node 脚本拦截 + 立即按文件拆分）

```bash
# 1) 校验仓库 & 分支
git rev-parse --git-dir            # 失败 → "当前目录不是 git 仓库"，停止
currentBranch=$(git branch --show-current)   # 空 → detached HEAD，停止
BASE_BRANCH=${BASE_BRANCH:-master}           # 默认 master；develop/release/xxx 等由用户指定
git branch -a | grep -E "(^|/)${BASE_BRANCH}$"   # 无匹配 → "基础分支不存在"，停止

# 2) 选 compareBase（优先远程，非阻塞 fetch）
git fetch origin "$BASE_BRANCH" 2>/dev/null || true
compareBase="origin/$BASE_BRANCH"   # 若 origin/$BASE_BRANCH 不存在则退回 "$BASE_BRANCH"

# 3) 未提交改动仅提示不阻塞
git status --porcelain             # 有输出 → ⚠️ 未提交改动不会进 diff，建议先 commit

# 4) 生成 diff（带 0.0 的 SCOPE pathspec）
# SCOPE 为空 → 不传 pathspec；SCOPE 非空 → 附加到 git diff 末尾
git diff "$compareBase...HEAD" -- $SCOPE > .code-review-diff.tmp
wc -c .code-review-diff.tmp        # 0 字节 → 删除文件，报"对应 SCOPE 下无改动，无需 CR"

# 5) 🔑 **立即按文件拆分（核心反爆炸机制）**
mkdir -p .cr-diffs && rm -f .cr-diffs/*.diff
awk '
  /^diff --git / {
    if (out) close(out)
    match($0, /b\/(.+)$/, m); safe = m[1]
    gsub(/[\/\\]/, "__", safe)
    out = ".cr-diffs/" safe ".diff"
  }
  out { print > out }
' .code-review-diff.tmp
ls .cr-diffs/ | sed 's/\.diff$//; s/__/\//g' > .cr-files.txt   # 纯文件清单，几百字节

# 6) 摘要（放 context，不进报告）
wc -l .cr-files.txt                           # 文件数
wc -l .code-review-diff.tmp                   # 总行数
du -k .code-review-diff.tmp | awk '{print $1}'   # 总大小 KB
```

生成成功后打印一行：`✅ Diff 已生成 & 拆分 | <currentBranch> vs <compareBase> | <n> 文件 / <n> 行 / <n> KB`

**🚫 严格禁止（会导致 context 爆炸，这是大 PR 卡死的首要根因）：**
- **禁止** `Read .code-review-diff.tmp`（整份 diff 可能几万行）
- **禁止**一次性把全部 `.cr-diffs/*.diff` 都读进 context

**✅ 正确做法：**
- 需要文件清单时 → `Read .cr-files.txt`（几百字节）
- 需要某个具体文件的 diff → `Read .cr-diffs/<safe>.diff`（单文件，几百行内）
- 审完一个文件 → 立即把结论写到覆盖矩阵，**丢弃该 diff 片段，不在 context 里保留**
- 如果某单文件的 diff 也超过 1000 行 → 用 `grep -A 20 -B 5 '^@@' .cr-diffs/<safe>.diff` 只取 hunks 头部 + 附近上下文

审查完成后 `rm -rf .code-review-diff.tmp .cr-diffs/ .cr-files.txt`。任何一步被拦截照**约定 A**处理。

### 0.2 拿仓库信息（用于拼可点击链接）

```bash
remoteUrl=$(git remote get-url origin)    # 失败 → 降级方案
currentBranch=$(git branch --show-current)
```

对 `remoteUrl` 试两条正则解析出 `org` 和 `repo`：
- SSH：`/^git@[^:]+:([^/]+)\/(.+?)(?:\.git)?$/`
- HTTPS：`/^https?:\/\/[^/]+\/([^/]+)\/(.+?)(?:\.git)?$/`

拼接：
```
codeBaseUrl = "https://dev.sankuai.com/code/repo-detail/${org}/${repo}"   # 美团内部平台
branch      = "refs/heads/${currentBranch}"
```

> **⚠️ 平台特化**：仓库 origin 指向 GitHub / GitLab / 其他内部平台时，替换成对应平台的"文件查看 URL 模板"，或直接降级为反引号纯文本格式。

**降级方案**（remote 失败 / 正则不匹配）：`codeBaseUrl=""`、`branch="refs/heads/master"`，报告里的位置字段改用反引号格式 `` `src/xxx.tsx:45` ``。**不阻塞 CR**。

### CR必做：改动意图收集（否则无法判断"该不该改"）

> **目的**：没有"意图"就审不出 G13（意图不明改动）和 G11（过度工程），因为你无法判断"这段代码是不是不必要的"。

```bash
# 1) PR 标题 + body（commit-msg 里往往写着"为什么"）
git log master..HEAD --pretty=format:'%h  %s%n%n%b%n---' | head -200

# 2) 最近 5 条 commit message（短小改动看最新几条就够）
git log -5 --pretty=format:'%h  %s'

# 3) 若仓库有 PR 链接，读 body（如 gh pr view --json body）
```

读完立刻在审查开始前输出 **"意图声明块"**（不得跳过）：

```
━━━ 本次 PR 改动意图声明 ━━━
核心目标：（从 commit/PR body 抽一句话，例如"给广告账户表单增加 DTO 提交字段"）
接口变更：（若有，列出后端接口签名新增/删除的字段；若无写"无"）
不在本次改动范围内：（例如"不涉及金额计算 / 不涉及权限判断 / 不重构现有组件结构"）
━━━━━━━━━━━━━━━━━━━━━━
```

⚠️ **找不到意图怎么办**（常见场景：commit message 是 "update"、"fix"、"修改代码"这类无信息量）：
1. 明确告知用户："commit / PR body 没有可读的意图说明，无法判断改动必要性"
2. 主动询问一句："本次 PR 的核心目标是什么？哪些接口变了？哪些文件**不应该**在本次改动范围内？"
3. 用户答复前**不得开始深度审查**（可以先跑第 2 步自动化检测，但第 3 步必须等意图齐全）

读取 diff：`Read .code-review-diff.tmp`，然后**根据 diff 规模选择审查模式**：

| diff 规模 | 审查模式 | 说明 |
|---------|---------|------|
| ≤ 200 行 | **标准模式** | 正常流程，一次性加载知识库，完整审查 |
| 200–600 行 | **分组模式** | 按文件类型分批处理：先审 .tsx/.ts，再审 .js，最后审配置文件 |
| 600–3000 行 | **两阶段模式** | 第一阶段仅用速查卡(G1-G8)快速扫描，标记 P0 候选；第二阶段逐个 P0 候选 Read 完整文件+加载对应知识库章节验证 |
| > 3000 行 | **多 Agent 模式** | 主 Agent 消化规则生成快照 → 并行启动 2-3 个 Sub-agent（每个携带规则快照 + 分组 diff）→ 主 Agent 合并去重 + 跨文件兜底 |

> **大 diff 核心策略**：先用口袋卡片（G1-G8）全局扫，速度快，噪声低；再对可疑点定点深挖。

**判定模式后初始化进度文件**（可选；后续各步按**约定 B** 执行 phase-start/phase-done，不再重复）：

```bash
# --mode 取值：standard / grouped / two-phase / multi-agent
node $SKILL_DIR/scripts/cr-progress.js init --mode <审查模式> --base <对比分支>
```

### 🔒 Context 续航（预防 + 补救，防止截断后带残缺记忆继续审查）

> **背景**：context 超长时，最早读取的内容（SKILL.md、知识库）会被截断。
> **两道防线**：前半"规则锚点"是预防（让规则始终在最近消息里），后半"自恢复"是补救（检测到信号就立即恢复）。

#### 预防 · 规则锚点（每组文件审查开始前必做）

在每一组文件审查**开始前**，先输出一个迷你规则确认块，把规则"刷"到 context 尾部：

```
━━━ 第 N 组审查（共 M 组）· 规则锚点 ━━━
本组文件：[文件列表]
P0 速查卡（本组有效）：G1/G3/G4...（只列与本组文件类型相关的条）
已加载知识库：[已读文件名]（不重复读取）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

这样无论 context 多长，当前组的规则始终出现在**最近的消息**里，不会因截断失效。

#### 补救 · 自恢复（检测到下列信号必须立即执行恢复动作）

| 感知信号 | 说明 | 恢复动作 |
|---------|------|---------|
| P0 三步验证步骤 C 失败（无法引用规则来源） | 最直接的截断信号，规则已丢失 | 重新 Read `SKILL.md`，锁定 G1-G13 后继续 |
| 不确定某类文件应该加载哪个知识库 | 加载表已丢失 | 重新 Read `SKILL.md` 第 1 步加载表部分 |
| 无法确认之前是否已加载某知识文件 | 加载记录已丢失 | 直接重新 Read 该知识文件（代价比猜测小）|
| 长轮次讨论后开始新文件组审查 | 规则可能已滑出 context 尾部 | 输出规则锚点块（不重读文件，只重申规则） |
| 用户说"你之前提到过……"但 AI 没有印象 | 历史分析已被截断 | 告知用户："我的上下文可能已被压缩，请确认你指的是哪条问题，我会重新核实" |
| 会话被打断 / 异常重启 | 整个审查状态丢失 | 若 `.cr-progress.json` 存在 → `Read` 恢复进度，跳到未完成的 phase 继续；不存在 → 告知用户需从头开始 |

**自恢复后的必要声明**（告知用户，避免静默错误）：

```
⚠️ 检测到上下文可能已被压缩，已重新加载 [SKILL.md / 知识文件名]。
   如有遗漏的历史结论，请告知，我会重新核查。
```

### 🤖 多 Agent 模式（diff > 3000 行）

> **为什么设 3000 行门槛**：多 Agent 能突破单 context 上限，但 Sub-agent 只有注入的规则，没有对话历史，跨文件感知弱。3000 行以下走两阶段模式更准，3000 行以上才值得付出协调成本。

#### Step M1：主 Agent 先消化所有知识文件，生成规则快照

> **执行顺序**：触发多 Agent 模式后，**先跳到第 1 步完整加载所有知识库**，再回到这里执行 M1-M4。知识库加载完毕才能提炼规则快照，Sub-agent 才有完整的规则可用。

**在启动任何 Sub-agent 之前**，主 Agent 必须完成第 1 步的所有知识文件加载，然后将关键规则提炼为一份**规则快照**（控制在 2000 token 以内）。

**规则快照的完整模板** → `$SKILL_DIR/knowledge/MULTI_AGENT_SNAPSHOT_TEMPLATE.md`

主 Agent 只需要照模板填入"【本次 diff 涉及的知识库摘要】"一节（按本次实际加载的知识库提炼规则名+识别特征），其余区块保持原样。

#### Step M2：按模块边界拆分，不按行数均分

拆分原则（有依赖关系的文件必须放同一组）：

```
优先按目录/功能模块拆：
  组 1：packages/container/（UI 层，页面+组件）
  组 2：packages/share/module/vm/（Store 层）
  组 3：packages/share/api/ + 配置文件（如有改动）

规则：
  - 同一文件不跨组拆分
  - A import B → A 和 B 放同一组
  - 每组 diff 不超过 1000 行
  - 最多拆 3 组（超过则合并小组）
```

#### Step M3：并行启动 Sub-agents，每个携带规则快照

每个 Sub-agent 的 prompt 模板同样放在 `$SKILL_DIR/knowledge/MULTI_AGENT_SNAPSHOT_TEMPLATE.md` 的"Sub-agent prompt 模板"一节，主 Agent 读取后把以下两个占位符填入实际内容，即为一个 Sub-agent 的完整 prompt：

- `[粘贴上方规则快照的全部内容]` → 第 M1 步生成的快照
- `[该组 diff 内容]` → 第 M2 步拆出来的本组 diff 文本

**落盘要求**：每个 Sub-agent 被要求**返回 JSON 时立刻写入 `.cr-sub-<N>.json`**（N = 组编号），不要只返回到内存。主 Agent 靠这些落盘文件判断谁跑完了、谁没跑完。

#### Step M3.5：健康检查 + 失败隔离（防一个 sub-agent 卡死拖垮整体）

> **背景**：单个 sub-agent 偶尔会卡（被模型 rate-limit、输入 token 超限、或在循环里反复 Grep 同一个 pattern）。过去一卡整体就没法继续，实际上其他组早已跑完。

**硬性策略**：
1. **分别起不同 Task，不串行等**：每个 sub-agent 各自独立的 TaskCreate，主 Agent 不能 `await Promise.all`，而是**轮询 `.cr-sub-*.json` 落盘情况**
2. **超时判定 = 15 分钟无落盘 + 无任何可见增量输出**（二者同时满足才算卡死，单一条件不算）
3. **累计超时 25 分钟** → 强制进入"容错决策点"，不再等待
4. 每个 sub-agent 的状态必须逐一登记到一张"M3.5 状态表"：

```
| 组 | 文件数 | diff 行数 | 状态 | 落盘 JSON | 备注 |
|---|------|---------|------|---------|------|
| 1 | UI 层   | 5 / 840  | ✅ 完成  | .cr-sub-1.json | 3 P0 / 2 P1 |
| 2 | Store 层 | 3 / 410  | ✅ 完成  | .cr-sub-2.json | 1 P0 / 4 P1 |
| 3 | API 层  | 2 / 120  | ❌ 超时  | —             | 15 min 无落盘无响应 |
```

**容错决策点（必须向用户明确提问，不允许 AI 自行替用户决定）**：

当检测到任一 sub-agent 超时 / 返回错误 JSON / 被用户中断时，主 Agent 必须输出以下提问：

```
⚠️ Sub-agent 执行状态：
  ✅ 组 1（UI 层）：已完成，发现 3 P0 / 2 P1
  ✅ 组 2（Store 层）：已完成，发现 1 P0 / 4 P1
  ❌ 组 3（API 层）：超时 15 分钟未响应（涉及文件：x.ts, y.ts）

下一步请选（回复 A/B/C）：
  [A] 先输出已完成部分（组 1+2），组 3 在覆盖矩阵标 "⚠️ 未审（超时）"，本次 CR 结论加"部分审查"标注
  [B] 主 Agent 接管重试组 3（顺序逐文件审，不再起 sub-agent，预计 5-10 分钟）
  [C] 把组 3 再拆成单文件 sub-agent 并行重试（适用于组内文件多且相互独立）
```

用户回复前**不要擅自进入 M4**。回复 A → 直接 M4 但在报告"📁 文件覆盖矩阵"节给组 3 文件标 `⚠️ 部分审查（sub-agent 超时）`；回复 B/C → 补完组 3 再进 M4。

**为什么必须问用户而不是默认 A**：用户可能着急出 CR（选 A），也可能不赶时间想拿完整结果（选 B/C）。两种需求都合理，不能 AI 替决定。

#### Step M4：主 Agent 合并 + 跨文件兜底

收到所有可用的 Sub-agent JSON 后（包括 M3.5 决策点里选 A 后只有部分组的情况），主 Agent 执行：

```
1. 去重：同 file + 相近 line（±3行）的问题只保留一条（取更高 P 级）
2. P 级一致性：同类型问题在不同组被判了不同级别 → 统一取高级
3. 跨文件验证：
   - 收集各组的 cross_file_hints
   - Grep 搜索相关调用方
   - 对有跨文件风险的问题补充说明或升级 P 级
4. **部分审查标注**（若 M3.5 选 A）：
   - 在报告"📊 整体评估"一句话结论里明写："本次为部分审查，组 X 未审（sub-agent 超时）"
   - 在覆盖矩阵里组 X 的文件状态写 `⚠️ 部分审查（sub-agent 超时）`
   - 合并结论前面加一句 "⚠️ 本报告基于 Y/Z 组 sub-agent 结果，结论可能不完整"
5. 输出最终合并报告（格式同标准模式）
```

> ⚠️ **多 Agent 模式的已知局限**：Sub-agent 只看自己那组 diff，无法感知跨组的业务语义。主 Agent 的跨文件兜底步骤必须执行，不可省略。

## 第 2 步：自动化检测

本步分两块：**安全扫描（AI 自己用 Grep 做，必做）** + **依赖/漏洞扫描（node 脚本，可选）**。

### 2.1 安全扫描（必做，AI 自己用 Grep 工具）

用 `Grep` 工具对本次 diff 涉及的 JS/TS 文件（保留 `.js`/`.jsx`/`.ts`/`.tsx`，**排除** `node_modules` / `dist` / `build` / `*.test.*` / `*.spec.*` / `*.stories.*` / `__mocks__` / `/api/` / `*.generated.*` / `*.auto.*`）扫下面 7 类模式：

| # | 模式 | 正则 | 级别 | 对应速查卡 |
|---|------|------|------|-----------|
| S1 | 硬编码 API Key / Token | `(api[_-]?key\|apikey\|access[_-]?token\|secret[_-]?key)\s*[:=]\s*['"\`][\w\-]{20,}['"\`]` | critical | G5 |
| S2 | 硬编码密码 | `password\s*[:=]\s*['"\`][^'"\`]{6,}['"\`]` | critical | G5 |
| S3 | 动态 innerHTML | `\.innerHTML\s*=\s*(?!['"\`])` | high | G4 |
| S4 | dangerouslySetInnerHTML | `dangerouslySetInnerHTML\s*=\s*\{\{?\s*__html:` | high | G4 |
| S5 | eval() | `\beval\s*\(` | critical | G4 |
| S6 | new Function | `new\s+Function\s*\(` | high | G4 变种 |
| S7 | console.log 敏感字段 | `console\.log\(.*?(password\|token\|secret\|key\|credential)` | medium | — |

**执行规则**：
- 扫描文件数 > 10 可一次性 `path: "."` + `glob: "**/*.{ts,tsx,js,jsx}"`，然后在结果里过滤 diff 涉及的文件；≤ 10 逐文件扫更精确
- 单个文件 > 500KB 跳过，输出 `⏭️ 跳过大文件: <path> (<size>KB)`
- 过滤后文件列表为空 → 输出"⚠️ 无变更的 JS/TS 文件，跳过安全扫描"，结束
- **critical / high 命中 → 第 3 步 P0 候选（S1/S2 对 G5；S3/S4/S5/S6 对 G4）；medium → P2 候选**（进候选后还要过第 4 步 Q1-Q5 核验）

**扫描结论写进报告"🤖 自动化检测结果"节**：`扫描文件数 N / 🔴 critical n / 🟠 high n / 🟡 medium n`，无命中写 `✅ 未发现安全问题`。Grep 工具失败时照**约定 A**降级（改用 `Bash grep -nE` 或人工 `Read` 全文扫描，不阻塞）。

### 2.2 依赖 & Bundle 扫描（可选，node 脚本）

```bash
# 首次使用先装依赖：cd $SKILL_DIR/scripts && npm install && cd -
node $SKILL_DIR/scripts/index.js
```

覆盖：幻影依赖、npm 漏洞、bundle 体积异常。

- ✅ 执行成功 → 在报告"🤖 自动化检测结果"节列出 index.js 的扫描结论
- ❌ **被权限拦截 / 依赖未装 / node 报错** → **跳过，不阻塞**。报告里写 `⚠️ 依赖扫描未执行（原因：权限拦截/未装依赖），安全扫描已由 2.1 覆盖`

> 2.1 已经独立覆盖了安全维度，2.2 失败不影响 CR 正确性。

## 第 3 步：深度审查

> **审查顺序**：P0 速查卡(G1-G13) → 意图对齐 → 必要性 5 问（审改动） → 知识库规则 → 影响范围评估
> 进度记录按**约定 B**执行；分组 / 两阶段 / 多 Agent 的 findings 文件按约定 B 表格落盘，防 context 截断丢失。

> **🔑 逐文件流式审查（反 context 爆炸硬约束）**：
> - 一次只处理 `.cr-files.txt` 里的**一个**文件；`Read .cr-diffs/<safe>.diff` 拿该文件的 diff 片段
> - 审完立即把结论写入**覆盖矩阵对应行** + 落盘到 `.cr-findings-<safe>.json`（防截断）
> - 然后**主动丢弃**该 diff 片段在 context 里的引用（下一个文件开始时，用"规则锚点"重刷规则，不要重复引用上一个文件的内容）
> - **禁止**：一次性 Read 多个 `.cr-diffs/*.diff` / Read `.code-review-diff.tmp` 整份 / 把已审文件的原始 diff 留在 context 里
> - 若单文件 diff 本身超 1000 行：先用 `grep '^@@' .cr-diffs/<safe>.diff` 拿所有 hunk 头，按 hunk 逐段 `Read` 带行号范围
> - 分组 / 两阶段 / 多 Agent 模式都遵守此约束；区别只在"谁来审这一个文件"（单 Agent 顺序 vs 多 Agent 并行）

### 3.0 必要性 5 问 N1-N5（审改动，强制门禁，针对 G11/G12/G13 —— 减法审查）

> **提示**：别和第 4 步的"核验 5 问 Q1-Q5"混淆。N 问审的是 **PR 代码本身**（"这段改动该不该存在"），Q 问审的是 **AI 自己的产出**（"这条发现是不是凑数"）。

> **触发范围**：对 diff 中**每个连续新增块 ≥ 5 行**、**每个新增的函数 / 组件 / 文件**，逐一回答下表 5 问。
> 这一步没做 = 本次审查没有做"减法"，必然漏过周航/董家力类评论。

| # | 问题 | 通过标准 | 不通过处理 |
|---|------|---------|-----------|
| N1 | **与 PR 意图对齐？** | 这段改动能回答 0.1 步声明的"核心目标"中的哪一条？若答不上且不解决任何已知 bug → G13 命中 | 挂 P1（G13），让 PR 作者给出理由；或建议整块撤销 |
| N2 | **能不能删？** | 把这段改动**整段删除**，功能是否仍然成立？（例：已有兜底的提前 return / 多余的 `Boolean()` 包裹 / 既加又删的等价抵消 / 可完全回滚的 export 重排） | 能删 → G11 命中，P1 建议删除 |
| N3 | **能不能更简？** | 能否用"三元 / 解构 / 既有工具函数 / 直接复用 DTO"达到同样效果？（例："这函数不需要，就一行：`row.xxx ? ${id}:${name} : '-'`"；"直接 `...item`"） | 可简 → G11 命中，P1 给出更简写法 |
| N4 | **与已有方案重复？** | 本地/仓库里是否已有同语义工具函数 / 类型 / 常量？（`Grep` 搜函数名/常量名/语义关键词） | 重复 → G11 命中，P1 改为复用 |
| N5 | **是否改变了接口契约？** | diff 是否改变了与后端的 DTO 结构？（`adAccountDTO` 的平铺/拼回、入参字段增减、字段类型 any→具体） | 若"是"且未对齐后端接口 / 未对齐意图声明中的"接口变更" → **G12 命中，P0** |

**执行规则**：
- **N2/N3 优先**：先做减法再做加法。任一回答"能删 / 能简"都必须输出，不得因为"代码能跑"就跳过
- N1 回答"意图不明"的整块改动 → 在报告中**集中列一个"🔎 意图不明改动清单"节**，让 PR 作者逐一解释
- N5 判 P0 时，必须同时 `Grep` 同仓库的 DTO / Store 方法，确认平铺/拼回对称性（参考 `MINIMAL_DIFF_PRINCIPLE.md` §三）
- N1-N4 本身不升级 P0（除非叠加 G1-G10 的正确性问题），但 N5 直接到 P0


**置信度原则**（核心防噪机制）：
- 报 P0：必须有代码行号 + 运行时后果 + 规则来源（见上方 P0 验证步骤 A/B/C）
- 报 P1：有代码证据，但影响不确定或有合理例外时，明确写出"证据是什么"
- 报 P2：有疑虑但证据不足时，标注 `(待确认)` 并说明疑虑点
- **禁止**：仅凭 diff 片段推测，或"这种写法通常有问题"这类无证据判断

**上下文扩展**（以下情况必须 Read 完整文件，不能基于 diff 片段决策）：

| 触发条件 | 扩展动作 |
|---------|---------|
| 改动了被多处调用的函数/Hook | `Grep` 搜调用方，抽样 Read 2-3 处 |
| 改动了 React 组件 props 接口或 state 逻辑 | `Read` 该组件完整文件 |
| 改动涉及金额/权限/状态流转核心逻辑 | `Read` 完整函数上下文 |
| diff 中出现类型引用但未见类型定义 | `Grep` 找类型声明 |

**始终检查（知识库之外的必查项）**：

| 类别 | 必查项 | 级别 |
|------|--------|------|
| 团队专项 | pnpm/yarn lock 文件是否大范围变动（>10行实质变化） | P0 |
| 影响范围 | 修改了已有函数/接口时，Grep 搜索全局调用方 | P1 |
| diff 删除行分析 | 所有 `-` 行必须逐条给出"删掉后的运行时语义差异"一句话；对含 `reset`/`clear`/`dispose`/`= undefined`/`= null`/`= []`/`= {}`/`runInAction` 内清零语句 等状态清理语义的删除行，必须额外说明"是否在同函数内等价替换"，无等价替换的按 G10 判 P0 | P0 |
| Store 生命周期对称性 | 文件命中"Store 类文件"触发（见第 1 步内容特征触发表）时，必须逐一检查 `open/init/show/reset/dispose` 五类方法是否对称：一个 observable 在 `open` 里被写，就必须能在 `reset`/`dispose`/下次 `open` 时被显式重置；条件分支赋值（`if/else`）必须覆盖所有分支 —— 未覆盖按 G9 判 P0 | P0 |

**diff 删除行分析的具体做法**（不可省略）：
```
1. 从 .code-review-diff.tmp 中 grep '^-' 拿到全部删除行（过滤掉 diff header 的 '---'）
2. 对每条 '-' 行，必须回答："这行删掉后，原来由它保证的运行时行为由谁接手？"
3. 若答案是"没人接手"或"不确定" → 升为 P0 候选（按 G10），并 Read 完整函数验证
4. 尤其关注 Store / Context / 全局单例类文件里的清理语句删除
```

## 第 3.5 步：违和感输出（强制，不查规则、不允许"无"）

> **为什么独立成一步**：规则层（G1-G13 / 知识库）只能抓**你已经见过的问题模式**。真实世界里最昂贵的 bug 往往长得不像任何已知模式——但人审一眼能觉出"怪怪的"。这一步的唯一目的是让 AI **绕开规则层，输出它自己的直觉信号**。
>
> **这一步不产出 P0/P1/P2**，它产出"问题候选线索"。如果线索里某条事后被第 4 步核验对上了具体规则，再正常走 P 级；对不上的，就以"违和感"形式留在报告里让 PR 作者确认。

### 硬性规则（不允许妥协）

1. **必须输出 3-5 条**，不允许出现"无违和感"、"未发现"、"已全部覆盖"这类回答。如果 AI 真觉得无感，就说"最不放心的 3 处"，总之必须列。
2. **不许引用任何 G1-G13 / M1-M14 / 知识库条款**。这一步就是要关掉规则检索，**逼 AI 说人话**。
3. **不许使用软措辞**（"可能有点"、"或许不太好"、"建议考虑"）。必须是"我觉得怪在【具体点】"的确定式表达。
4. **每条必须锚定到具体文件:行号**，禁止"整体感觉太复杂"、"整体命名不够好"这类空话。
5. **Top-3 必须是跨文件 / 跨层的观察**，不能全是单文件内部小毛病。这一条专治"只盯 diff 行不看整体"。

### 输出格式（严格）

```
🧐 AI 违和感清单（必填 3-5 条）

1. [文件路径:行号] —— 怪在：（一句话，≤30 字）
   解释：（一句话，为什么怪 + 建议 PR 作者澄清什么 / 或 AI 推测的可能问题）

2. ……
```

### 判断"什么算怪"的启发（仅供参考，不是清单）

- **三方不一致**：接口声明 A，实现写 B，调用方按 C 用（如 `T | Promise<T>` union 的接口 + 纯同步实现 + 总是 `await` 的调用方）
- **抽象层次跳跃**：UI 组件里直接拼 SQL / 调 Redis key / 读全局单例
- **命名 ≠ 行为**：函数叫 `validateX`，里面实际在 mutate state
- **结构重复但不一致**：两个相邻分支做类似的事，写法却不同
- **防御性过度 / 防御性不足**：某些路径 `?.?.?.?.`，同文件另一处直接 `.x.y.z`
- **Diff 中的"消失"**：原代码有但没出现在 diff 里的重要能力（如 loading / error / 日志），要问"为什么新代码不需要了"
- **魔数 / 常量直写**：`0` / `1` / `-1` / 硬编码 URL / 硬编码时长
- **同步 / 异步错配**：`await` 一个同步函数 / 异步函数没 `await` 当同步用
- **类型收紧后的残留**：原本 `any` 现在收紧成 `User`，但内部某个 cast / `as any` 没清干净

> 注意：这份列表**只是训练你的嗅觉**，不是要你逐条匹配。最好的违和感是不在任何列表里的那一条。

### 执行约束

- 不要用这一步替代规则层审查（第 3 步仍然必做）
- 不要用规则层结果替代这一步（即使 P0/P1/P2 已经写了 5 条，违和感清单还是要独立输出）
- 重复：一条违和感如果事后被 Q1-Q5 核验升级成 P0/P1/P2，**它仍然保留在违和感清单**（标注"→ 已升级为 P0-N"），不要删，这样能沉淀"AI 直觉 ↔ 规则命中"的映射，供后续 skill 迭代参考

## 第 4 步：核验 5 问 Q1-Q5（审自己）+ 价值过滤（强制门禁，不可跳过）

> **区分**：N 问（3.0）审 **PR 代码**，Q 问（4.1）审 **AI 自己给出的 P0/P1/P2 草稿**。目的不同，别弄混。

> **进度记录**：除了约定 B 的 `phase-start/phase-done verification_filter`，结束后还要跑 `update-counts --p0 N --p1 N --p2 N` 把核验后的最终条目数写回 `.cr-progress.json`（约定 B 尾行有提示）。

> **目的**：第 3 步会产出"问题候选清单"（P0/P1/P2 草稿），本步负责**把假阳性、吹毛求疵、凑数条目全部清掉**，并保证每条真问题证据链完整。
>
> **执行次序**：先走 **4.1 五问核验**（杀假阳性）→ 再走 **4.2 价值过滤**（治信噪比）→ 再走后面的 🚦 五项工程自检门禁。三者都通过才能进入 `输出报告`。
>
> **关键纪律**：本步产出的草稿**不要输出给用户**；只有最终通过的条目才进入报告。

### 4.1 每条问题候选必过的"核验 6 问 Q0-Q5（审自己）"

对草稿里**每一条 P0 / P1 / P2** 依次回答下表 6 个问题，任何一项不通过就按规则处理：

| # | 问题 | 通过标准 | 不通过处理 |
|---|------|---------|-----------|
| **Q0** | **跨文件扫描**（仅当问题涉及"改动可被外部依赖的接口"时触发）| 触发范围：①**导出函数 / Hook / 组件 props / 类型 / 常量** 的签名变更；② **DTO / 接口字段**的新增/删除/类型变化；③ **Store 的 observable** 新增/删除/语义变化。触发后必须附带 **Grep 命令 + 命中行数 + 至少 1 处抽样调用点 Read 结果**，且在条目末尾的 `跨文件验证` 字段里登记 | **P0/P1 直接降 P2（待确认）**；同时在报告里标注"跨文件验证未做，无法断言 P0/P1"，Q0 绝不可"略过"—— 略过即视为假阳性候选 |
| Q1 | **行号核验** | 用 `Read` 工具实际打开该文件的该行，看到的内容与"问题代码"片段一致（或片段是从该区域原样摘录） | **直接删除该条**（禁止保留"凭印象"的条目） |
| Q2 | **后果具体化** | 能写出三要素："**① 触发条件**（什么场景/什么输入会发生）+ **② 可观察现象**（用户/监控/日志看到什么）+ **③ 影响量级**（全量用户 / 特定路径 / 边缘 case，或影响的 RPM / 请求比例）"。不能是"可能有问题"、"可读性下降"、"降低可维护性"、"增加心智负担"这类模糊词 | P0/P1 → 降级 P2(待确认)；P2 → 删除 |
| Q3 | **规则原文** | 条目末尾 `规则原文` 字段里**真实摘录**了一段 ≤50 字、能支持本条判断的依据文本（可来自 G1-G13 速查卡、知识库某章节、或团队规范原句）。**字段为空 / 仅填"见 G3"之类标签 → Q3 不通过** | P0 → 降级 P1；P1 → 降级 P2；P2 → 删除 |
| Q4 | **反向自问** | 自问一次："如果我是 PR owner 看到这条，会觉得是 AI 在凑数吗？"。答案是"会" | P1/P2 → 删除；P0 可保留但必须补强 Q2（否则降级 P1） |
| Q5 | **成本标注** | 明确写出修改成本：`1 行` / `1 函数` / `跨文件` / `需重构`。成本与收益严重不匹配（例：为一条 P2 建议"全局重构"）| 删除该条 |

> **Q0 特别说明**：这一步是第 3 步"上下文扩展"表的"必做版"——第 3 步只是建议在可疑处搜调用方，Q0 是在 P0/P1 草稿出来之后**强制核对**。涉及接口/契约变更但没搜调用方，就默认它是"局部看起来错了但整体可能是对的"，不得升到 P0/P1。

**执行步骤（按顺序做完再进入 4.2）**：
```
for 每一条候选问题 in (P0_草稿, P1_草稿, P2_草稿):
    依次回答 Q0 → Q5（Q0 不触发时直接跳到 Q1）
    记录本条的"核验结论"（通过 / 删除 / 降级为 Px）
    若降级：将条目移动到对应级别的草稿列表
    若删除：从草稿中移除，不保留痕迹
```

> **Q1 特别说明**：对同一文件的多个问题，`Read` 一次文件即可复用；但**禁止**跳过 Q1 直接"凭 diff 片段"确认，因为 diff 只显示改动行及少量上下文，行号可能错位。

### 4.2 价值过滤 · 信噪比门禁

走完 4.1 后再做全局过滤：

| 规则 | 阈值 / 要求 | 超标处理 |
|------|------------|---------|
| **P1 条目数上限** | ≤ 5 条 | 按"不修后果严重度 × 1/修改成本"排序，保留 Top 5，其余降级 P2 或并入"另外 N 处同类" |
| **P2 条目数上限** | ≤ 8 条 | 同上，超出直接删除 |
| **同类合并** | 同一规则在 ≥3 个文件触发 | 合并为 1 条，末尾附"另外 N 处同类：file1:line, file2:line, …" |
| **纯风格类限流** | 命名 / 注释 / 格式类问题在整份报告 ≤ 2 条 | 超出只保留影响最大的 2 条 |
| **"建议"禁用** | 任何条目描述里不得出现"建议考虑"、"可以尝试"、"也许可以"等软措辞 | 改写成"【修改动作】：……"或删除 |
| **P0 不设上限** | 安全 / 正确性问题必须全部保留 | —— |

**4.1 + 4.2 执行完毕后，必须在报告 `📋 上下文分析` 节追加一行**：
```
📊 核验汇总：候选 X 条 → 核验后 Y 条（删除 A 条 / 降级 B 条 / 合并 C 条）
```
这一行不存在 → 视为跳过第 4 步，必须重做。

### 4.3 每条问题强制字段（与报告模板对齐）

4.1 + 4.2 通过的每条问题，进入报告时必须同时具备：
- `位置` `依据` `规则原文` `运行时后果` `bad code` `fix code`（P0 + P1 全部必填；P2 至少有 `位置` + `依据` + `待确认说明`）
- `不修后果`：Q2 三要素——**触发条件 + 可观察现象 + 影响量级**（全量 / 特定路径 / 边缘 case）
- `修改成本`：`1 行` / `1 函数` / `跨文件` / `需重构`（Q5 产物）
- `跨文件验证`：**Q0 触发时必填**（Grep 命令 + 命中数 + 抽样调用点）；未触发写 `—`
- `规则原文`：**Q3 产物，必填真实摘录**（≤50 字），不允许"见 G3"这种标签式写法
- 缺任一字段 → 不得输出，回到 4.1 重补

## 输出报告

> ⛔ **严禁**：输出聊天摘要、bullet-point 总结、"需要我帮你修复吗？"这类对话式结尾。
> ✅ **必须**：按以下结构输出完整的结构化报告，每节都必须存在（即使内容是"不涉及"）。
>
> **分批输出规则**（分组/两阶段模式下）：每组审查结束后先输出该组的子发现列表（格式同正式报告的 P0/P1/P2 节），
> 所有组完成后再输出合并的"整体评估"节。这样可避免一次性输出超长报告本身消耗大量 context。

### 报告必须包含的节（按顺序）

```
### 📋 上下文分析
  - 本次 PR 改动意图（从 0.1 步意图声明中誊抄一句话）
  - 已读取的文件列表 + 关键改动说明
  - **影响范围（Grep 搜索结果，必须贴命令+命中数+抽样调用点）**：
    - 格式：每条占一行 —— `Grep pattern='<正则>' glob='<范围>' → N 处命中，抽样 <file>:<line>`
    - 只写"已 Grep 影响范围"这种结论性描述 → 视为跳过扫描，本节不通过
    - 未触发 Q0（即本次 PR 无导出函数/DTO/Store observable 变更）→ 明确写 "无 Q0 触发项"，不要留空
  - 📊 核验汇总：候选 X 条 → 核验后 Y 条（删除 A / 降级 B / 合并 C）   ← 第 4 步产物，必填

### 📁 文件覆盖矩阵（必填，所有 diff 文件逐一点名）
  - **生成方式**：`Read .cr-files.txt` 拿到纯文件清单（禁止 Read `.code-review-diff.tmp` 整份），然后对**每个**文件跑 `awk '/^\+\+\+/||/^---/{next} /^\+/{a++} /^-/{d++} END{print a,d}' .cr-diffs/<safe>.diff` 拿到 +/- 行数；`src !== dst` 时记为 `<src> → <dst>`
  - **表头**：`# / 文件 / 新增 / 删除 / 审查状态 / 问题数`；初始状态 `❌ 未审（必须补）`、问题数 `—`；按 `.cr-files.txt` 顺序排列
  - **状态四选一**：`✅ 已审（N 条问题）` / `✅ 已审（无问题）` / `⚠️ 部分审查（说明未覆盖部分）` / `❌ 未审（必须补）`
  - ❌ 存在任何一行 → 本次 CR 失败，禁止输出最终结论，必须补审后重跑
  - 多 Agent 模式下：主 Agent 合并完 sub-agent 结果后**必须**和本矩阵交叉核对，sub-agent 没审到的文件必须由主 Agent 补审

### 🤖 自动化检测结果
  - 安全扫描（2.1，必做，基于 Grep）：✅ 通过 / ❌ 问题描述（critical/high/medium 计数）
  - 依赖扫描（2.2，可选，index.js）：✅ 通过 / ❌ 问题描述 / ⚠️ 未执行（原因：权限拦截 / 未装依赖 / node 报错）

### 每个问题条目的标准格式

```markdown
**[Px-N] 问题标题**
- **位置**：[文件路径:行号](可点击链接)
- **依据**：速查卡 G3 / TYPESCRIPT_STANDARDS.md §二
- **规则原文**：（**真实摘录**，≤50 字，P0/P1 必填；禁止写"见 G3"这种标签式引用）
   例："catch 块内不得为空；吞掉异常是 P0。" —— SKILL.md §P0 速查卡 G3
- **不修后果**：（Q2 三要素，P0/P1 必填）
   ① 触发条件：（什么场景/输入会发生）
   ② 可观察现象：（用户 / 监控 / 日志看到什么）
   ③ 影响量级：（全量用户 / 特定路径 / 边缘 case / 影响比例）
   例：① 用户网络抖动导致 KNB.getLocation 超时 ② "我的位置"按钮 loading 永不消失，需手动刷新 ③ 全量 iOS 用户首次定位路径（约 35% DAU）
- **修改成本**：`1 行` / `1 函数` / `跨文件` / `需重构` 四档之一
- **跨文件验证**：（Q0 触发时必填，否则写 `—`）
   例：`Grep pattern='adAccountDTO' glob='**/*.{ts,tsx}' → 7 处命中，抽样 src/pages/ad/form.tsx:45 已 Read 确认需同步改`
- **问题代码**：
  ```js
  // ❌ 当前写法
  ```
- **修复方案**：
  ```js
  // ✅ 修复后
  ```
```

P2 条目格式（更简）：
```markdown
**[P2-N] 问题标题**
- **位置**：[文件路径:行号](可点击链接)
- **依据**：（规则来源）
- **待确认说明**：（疑虑点 + 需要核实什么才能定级）
- **修改成本**：`1 行` / `1 函数` / `跨文件` / `需重构`
```

> **合格门禁**：没有 `位置` + `依据` + `规则原文`(真实摘录) + `不修后果`(三要素) + `修改成本` + `跨文件验证`(Q0 触发时) + 代码片段的条目不是合格的 CR 输出。第 4 步会强制拦截，不合格的条目会被删除或降级。

### P0 / P1 常见误判纠正

| 常见误判 | 正确级别 | 依据 |
|---------|---------|------|
| 空 catch 块 / catch 后直接 throw | **P0**（非 P1） | 速查卡 G3 |
| `as unknown as X` 双重断言 | **P0**（非 P1） | TYPESCRIPT_STANDARDS.md §二 |
| `new Promise` 缺 else 分支 | **P0**（非 P1） | 速查卡 G1 |
| `if (count)` 当 count 可能为 0 | **P0**（非 P1） | 速查卡 G2 / JS §二.1 |
| useEffect 依赖数组不完整 | **P1**（不是 P0） | REACT §一.1 |
| 命名不规范 / 缺注释 | **P2**（不是 P1） | 规范类问题 |

## 审查完成后

```bash
rm -rf .code-review-diff.tmp .cr-diffs/ .cr-files.txt .cr-findings-*.json .cr-group-*.json .cr-sub-*.json
node $SKILL_DIR/scripts/cr-progress.js cleanup   # 可选
```

> 若本次审查因异常提前结束且用户可能继续追问（如二次审查、局部修订）→ **先不 cleanup**，保留 `.cr-progress.json` / `.code-review-diff.tmp` / `.cr-diffs/` / `.cr-findings-*.json` 供下次恢复。只有"审查彻底完成、无需再回到本次 diff"时才清理。
> `rm` 被权限拦截时，告诉用户"请手动删除这些中间文件"，**不要反复重试**。
