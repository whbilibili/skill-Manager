# 基础研发平台 | C++编码规范

# 一、原则

【强制】编写遵从ISO标准的Modern C++代码。

【建议】用代码直接表达想法。

【建议】面向读而非写编程，但在必要时为优化让路。

【建议】不过度设计。

【建议】宁要显式，不要隐式。

【建议】避免使用奇特或危险的语法结构，避免使用那些正常水平的C++工程师认为棘手或难以维护的高阶特性。

【强制】按一致的风格编程。

【建议】尽可能早的捕获错误，广泛的使用断言。

---

# 二、规范

## 1、变量

【建议】就近定义，不要在函数开头集中定义局部变量（过时的做法）。

【建议】尽量在小的范围内声明变量。

【强制】定义变量的同时赋初值。

【建议】对于全局变量的名称添加g_作为前缀。

【建议】尽量少的使用全局变量。

【强制】禁止在头文件定义static全局变量。

【建议】慎用thread_local，优先考虑使用函数参数、局部变量或者类成员变量来传递和存储线程特定的数据。

## 2、表达式和语句

【强制】一行一个语句，不要将多个语句写在同一行。

【建议】根据数据类型的不同，使用不同的判空判等方式。

【强制】整型判空： if (variable == 0) 或者 if (variable != 0)

【强制】超过3个的条件表达式，必须显式使用圆括号，而不是靠猜判断优先级。

【建议】优先使用基于范围的循环。

【建议】总是为switch提供default，勿忘每个break。

## 3、函数

【建议】偏好纯函数。

【建议】一个函数聚焦于单一逻辑。

【建议】如果一个操作可被复用，则给它一个名字。

【建议】考虑用标准库算法替代手写逻辑。

【建议】函数行数不要超过一屏为宜，否则考虑拆分。

【建议】代码尽量自解释，不写无用的注释。

【建议】参数数量不要太多，最好不超过6个。

【强制】参数顺序：先写输入参数，再写输出参数。

【强制】只允许在非虚函数中使用默认参数，且默认参数的值始终一致。

【强制】不要通过单指针参数传递数组。

【强制】对于输入参数：built-in类型传值，其他类型传const引用。

【建议】优先使用返回值做输出；如果要用参数做输出，用于输出的参数定义在参数列表的后部。

## 4、重载和覆盖

【建议】函数重载含义要明确，要让读代码的人不花心思即可猜测调用的是哪个版本。

【强制】只有意义明显、不会出现奇怪的行为并且与对应的内建运算符的行为一致时才重载运算符。

【强制】重载操作符的时候要注意配对。

【建议】尽量不要重载全局的operator new和operator delete。

【强制】不要重载二元运算符&&和||，也不要重载一元运算符&。

【强制】子类需显式override基类虚函数。

## 5、类、结构体和枚举

【建议】一个类应聚焦于一个职责，不要巨类、杂类。

【建议】定义类的时候，偏好值类型。

【建议】优先使用class定义类型。

【建议】构造函数只做成员变量初始化和资源分配等简单操作。

【强制】构造函数的初始化列表不要乱序。

【强制】使用explicit关键字定义单参数构造函数。

【建议】析构函数建议只做资源释放等简单操作。

【强制】抽象基类的析构函数要加virtual关键字。

【建议】子类的析构函数如无额外逻辑，无须显式定义。

【强制】不要在构造函数和析构函数中调用虚函数。

【强制】如果你的类不支持复制/移动操作，请显式禁用。

【强制】不要为类提供隐式类型转换函数。

【强制】不要公开类的成员变量，结构体除外。

【建议】成员函数内联无须加inline。

【建议】在定义成员变量的同时，以一致的方式做初始化。

【强制】不修改内部状态的只读函数必须声明为const。

【建议】使用强类型枚举enum class替代enum。

【建议】理解和遵从类定义的3/5/0法则。

## 6、类型转换

【建议】优先使用C++风格的转型而非C风格的转型。

【建议】不要使用dynamic_cast。

【建议】不要使用const_cast。

## 7、接口和继承

【建议】区分接口和实现，不要随意公开类的成员函数。

【建议】面向接口而非面向实现编程。

【建议】控制接口数量，最小化暴露成员。

【建议】当你不希望类被继承时，应使用final关键字。

【强制】只在真正期待子类override的接口，才应该被定义为虚函数。

【强制】子类不要覆盖父类的非虚函数。

【建议】慎用多重继承、警惕菱形继承。

【强制】确保公有继承模塑的是is-a关系。

【强制】不要从具象类（可实例化的类）派生。

【建议】组合优先于继承。

## 8、模板和泛型编程

【建议】应用开发中要慎用模板和泛型编程，这是本规范关于模板和泛型编程的一个主原则。

【建议】只在非常必要的情况下才使用模板元编程。

【强制】不要特化函数模板。

【强制】不使用类模板参数推导（C++17以上）。

【建议】优先使用using而非typedef定义类型别名。

## 9、宏和预处理

【建议】除非用于条件编译、跟踪调试，或者能够显著减少代码量并确保可读性，否则不用宏。

【建议】如果必须编写宏函数，避免引入控制语句，如break，continue，return等。

【强制】如果必须编写宏/宏函数，避免宏/宏函数依赖全局变量。

【强制】宏代码尽可能的用 do...while(0) 块包围。

## 10、名字空间

【强制】确保代码置于项目私有namespace保护之下。

【建议】名字空间嵌套最多2层。

【建议】将辅助函数与其支持类置于同一名字空间下。

【建议】用匿名名字空间替代static全局变量、static全局函数的用法。

【强制】不要在头文件using namespace xxx。

## 11、文件和目录

【建议】一个头文件包含一个主要类，头文件名与主要类对应。

【强制】一个项目的头文件和源文件不要混用多种后缀。

【建议】一个源文件尽量和一个头文件对应。

【建议】优先使用前向声明，而非头文件包含。

【建议】使用统一的方式防头文件重复包含。

【建议】头文件包含遵从"自包含原则"。

【建议】同名的源文件和头文件应放在同一个目录下。

【建议】C++项目的目录不要嵌套过深。

## 12、命名约定

【建议】遵从命名的一般性原则。

【建议】给变量一个简短且有意义名字。

【强制】变量一律小写，单词之间用下划线连接。类的成员变量以下划线结尾，但结构体成员无须下划线结尾。

【建议】局部变量命名在不增加理解困难的情况下，可以简短，也可以缩写。

【建议】给函数起一个好名字。

【建议】无须将参数信息编码进函数名。

【建议】类型名称（包括类名和结构体名）建议用名词、名词短语。

【强制】类型名称的每单词首字母大写，不包含下划线。

【强制】枚举的命名与常量一致，都以k打头。

【建议】文件名全小写，单词之间用下划线连接。

## 13、资源管理

【建议】原始指针（T*）意味着"不拥有"。

【建议】原始引用（T&），意味着"不拥有"。

【强制】禁止使用auto_ptr。

【建议】动态分配的对象，考虑立马用智能指针管理起来。

【建议】一个表达式语句最多只显式分配一个资源。

【建议】优先使用作用域对象，非必要不进行堆分配。

【建议】如果需要传递对象，使用std::unique_ptr明确所有权传递。

【建议】如需共享所有权，使用std::shared_ptr明确所有权共享。

【建议】使用资源句柄或RAII自动管理资源。

【建议】偏好unique_ptr而非shared_ptr，除非真的需要共享所有权。

【建议】使用std::make_unique去创建独占智能指针，使用std::make_shared去创建共享智能指针。

【建议】遵从指针参数、智能指针参数的潜规则。

## 14、错误处理和异常

【建议】不使用异常，除非已有项目/底层库使用了异常，这时候须catch所有异常。

【建议】推荐用返回值报告错误，而非抛异常。

【强制】如果必须使用异常，绝不让异常离开析构函数。

【强制】禁止使用未显式指明目标缓冲区大小的字符串操作函数，请以strcpy_n替代strcpy。

【强制】delete、free之后，立即对指针赋空。

【建议】优先使用编译期错误检查，而不是运行时错误检查。

【强制】对标准库容器遍历的同时做删除，要谨防迭代器失效。

## 15、并发和同步

【建议】优先编写线程安全的函数。

【强制】不要在函数里定义大的临时变量（buf或者数组）。

【建议】优先使用C++标准库thread而非Posix线程。

【建议】最小化线程创建/销毁，最小化线程上下文切换。

【建议】最小化持有锁的时间。

【建议】优先使用RAII而非显式lock()/unlock()。

【建议】使用std::lock()或std::scoped_lock()去申请多个mutex。

【建议】优先使用mutex而非其他同步机制（memory order、std::atomic、读写锁、lock-free、条件变量等）。

【建议】绝不在持有锁的时候调用未知行为的函数。

【建议】谨慎设计锁的粒度。

## 16、标准库和三方库

【建议】C++标准库优先于其他三方库。总是优先使用C++标准库而非boost、folly、abseil等三方库。

【建议】不要对标准库做侵入式扩充（比如在namespace std下加东西），不要试图从标准库容器或类型派生。

【建议】对引入三方库保持警惕和克制。

## 17、性能

【建议】正确性和可读性优先。

【建议】性能优化原则。

【建议】避免过早优化和过度优化。

【建议】临时字符串处理考虑用std::string_view替代std::string。

【建议】理解右值引用，恰当的使用move语义提升性能。

## 18、不为清单

【建议】不要使用pimpl等小伎俩。

【建议】不使用placement new和显式析构。

【建议】尽量不使用bitfield、内嵌汇编。

【建议】不滥用mutable变量。

【建议】不要使用成员函数指针、成员变量指针。

【建议】不要使用运行时类型识别：RTTI。

【建议】慎用C的可变参数和C++可变模版参数。

【建议】允许在合理范围内使用友元，但要慎用友元。

## 19、代码格式

### 行长度

【强制】每一行代码字符数不超过120。

### 换行 / 断行

【强制】"{" 前不断行。

【强制】异常处理的catch前不换行。

【强制】条件语句if / else if / else前不换行。

【强制】do while / while关键字前不换行。

【强制】属性列表 attribue，[[]] 后不换行。

【强制】模板 template <typename T> 后总是换行。

【强制】概念concept后不换行。

【强制】构造函数初始值列表，如需换行，总是在冒号":"前换行。

【强制】多重继承列表，如需换行，总是在":"前换行。

### 缩进

【强制】只使用空格，不使用制表符。

【强制】访问控制符缩进1格空格，列表初始化缩进4格空格，其他缩进2空格。

【强制】预处理指令不要缩进，顶格写。

【强制】命名空间内的函数/变量定义顶格写，不缩进。

【强制】requires(C++20关键字)表达式另起一行，不要缩进，顶格写。

【强制】函数返回类型换行时，不缩进函数名。

### 列表初始化

【强制】如果列表初始化前伴随着名字（比如类型或变量名），则将名字视作是函数调用的括号来格式化。反之，就当它有个长度为零的名字来格式化。

【强制】C++11 列表初始化，连续使用列表初始化时连续写"{{"，外层与内层"{"间不加空格。

### 二元 / 三元操作数

【强制】二元 / 三元表达式尽可能写在一行，如需拆分多行，将单个操作数垂直方向进行对齐。

【强制】三元表达式操作符写在行首。

### Lambda 表达式

【强制】Lambda 表达式对形参和函数体的格式化和其他函数一致，捕获列表同理，表项用逗号隔开。

【强制】短的Lambda函数可以写在一行。

【强制】相对于Lambda签名对齐Lambda主体。

### 指针和引用表达式

【建议】指针声明符*和引用声明符&，可以靠近类型也可以靠近变量名。

【强制】const、volatile、inline、constexpr、static、friend、restrict 等类型修饰符靠左与类型结合或齐。

### 转义换行符

【强制】尽量向左对齐转义换行符 "\"。

### 条件语句

【强制】所有情况下if / else if / else和左圆括号间都有个空格，右圆括号和左大括号之间也要有个空格。

【强制】在圆括号和条件之间没有空格。

【强制】如果语句中某个if-else分支使用了大括号的话，其它分支也必须使用。

### 循环

【强制】所有情况下while / for和左圆括号间都有个空格，右圆括号和左大括号之间也要有个空格。

【强制】在圆括号和条件之间没有空格。

【强制】空循环体应使用{}或continue，而不是一个简单的分号。

### Switch

【强制】case语句缩进2个空格。

### 类格式

【强制】访问说明符控制块的声明依次序是public, protected, private。

【强制】可多次出现相同的访问说明符。

【强制】访问说明符缩进1个空格。

【强制】当访问说明符开始一个新的逻辑块时才在访问说明符上方添加空行。

【强制】构造函数初始化列表尽可能与构造函数名写在一行，如果一行写不下，4 格缩进后写在下行，如果下一行也空间不够，则每行写一个成员。

### 函数声明与定义

【强制】只有在参数未被使用或者其用途非常明显时，才能省略参数名。

【强制】如果返回类型与函数声明或定义分行了，不要缩进。

【强制】左圆括号和函数名间没有空格，在同一行。

【强制】左大括号总在最后一个参数同一行的末尾处，不另起新行。

【强制】右大括号总是单独位于函数最后一行，或者与左大括号同一行。

【强制】右圆括号和左大括号间总是有一个空格。

【强制】换行后的参数保持 4个空格的缩进。

### 函数调用

【强制】要么一行写完函数调用，要么在圆括号里对参数分行，要么参数另起一行且缩进四格。

【强制】如果同一行放不下，可断为多行，缩进四格，后面每一行都和第一个实参对齐，左圆括号后和右圆括号前不要留空格。

【强制】如果一系列参数本身就有一定的结构, 可以酌情地按其结构来决定参数换行位置。

### 模板

【强制】template <typename T>模板声明后总是换行；template关键字与<typename T>之间空格。

### 名字空间

【强制】嵌套名字空间声明写在多行，每行定义一个名字空间。

### 头文件

【强制】优先使用std名字空间下的C++头文件，如使用<cstdio>代替<stdio.h>。

【强制】避免使用文件路径"."和".."。

【强制】相同类别的头文件合并在一起并按照字典序内部排序，不同类别的头文件间使用空行分隔，头文件类别按下面优先级排序。

### 注释

【建议】代码本身能直白传递的信息，不额外注释；有特殊考虑的代码，需加注释。

【建议】接口需要加注释，写清楚接口作用，调用约束和各参数含义。

【强制】连接行末尾的注释，遵循统一的风格，要么垂直对齐，要么与行末尾保持相同的空格数。

## 20、信息安全

【强制】安全使用创建进程类函数。

【强制】根据文件的敏感级别设置不同的访问权限，防止敏感数据被恶意程序访问。

【建议】弱加密、弱哈希、弱随机、不安全的协议等相关库、函数、类、宏、常量等不应出现在代码中。

【建议】使用相对路径可能导致一些安全风险，例如DLL、EXE劫持等问题。

【建议】注意有无符号整型强转导致的逻辑错误。

【建议】需要考虑整型溢出的可能。

【建议】不要开启"掩盖错误、不符合标准、屏蔽安全措施"等不安全编译选项。

【建议】敏感数据请遵循以下原则避免泄露。

## 21、单元测试

【建议】选用易于使用、跨平台的单测框架，如gtest。

【建议】选用可以与单测框架无缝集成并具备强大模拟功能的Mock工具，如 gmock。

【强制】一个测试函数只测试一个函数。

【强制】单元测试要保持独立。

【强制】单元测试必须要有逻辑验证能力和强度。

【强制】单元测试应当避免连接数据库或网络。

【强制】测试数据需要具有差异性，能够覆盖边界条件和异常输入。

【强制】保证你的测试是幂等的。
