# 基础研发平台 | Python编码规范

### 一、原则

#### 1、用代码直接表达想法

保持代码简洁清晰，确保逻辑直观易懂，避免引入难以理解的机制和副作用。

#### 2、重视代码可读性

类、变量、函数的命名应该贴近自然语言，避免出现与直观理解不一致的运行逻辑。

#### 3、尽可能早的捕获错误

尽可能明确接口、变量的数据类型和数值范围，根据具体错误抛出意义明确的异常。不使用过于灵活的接口设计和宽泛的异常处理。

#### 4、最小权限、最小功能、最小暴露

简化并拆分代码功能，保持每个函数的功能尽量单一，函数接口应使用基础数据类型，便于实现单测。

#### 5、使用Python3

本规范参考了PEP 8和Google Python规范内容，面向Python3环境，Python2不提倡继续使用。Python的语法功能以及安全特性保持活跃更新，鼓励向稳定版的新特性迁移。

### 二、编码规约

#### 1、代码风格

1). 【强制】缩进采用4个空格，缩进代码段不要使用制表符或者混用制表符和空格。

2). 【强制】合适的地方使用空格：

- 括号()、[]、{}内部不要有多余的空格。
- 不在逗号、分号、冒号前加空格，加在后面（除非在行尾）。如果逗号后面是右小括号，不要加空格。
- 切片操作中冒号前后不加空格，除非起止位置或者步长位置中包含函数或者运算操作。
- 在参数列表、索引、切片括号前不加空格。
- 不要为了使语句对齐在赋值（或其他）运算符周围使用多个空格。

3). 【强制】正确使用代码注释：

- 块注释一般写在对应代码之前，并且和对应代码有同样的缩进级别。块注释的每一行都应该以#和一个空格开头。
- 行内注释是和代码语句写在一行内的注释。行内注释应该至少和代码语句之间有两个空格的间隔，并且以#和一个空格开始（尽量不要行内注释）。

4). 【建议】文档字符串使用三重双引号`"""`，多行文档字符串以单行`"""`结尾。注释内容应该包括：

- 第一行是句号、问号或惊叹号结尾的摘要。如果是多行注释，第二行是空行。
- 输入参数，复杂参数应给出示例。
- 返回数据类型和语义信息，除非返回 None。

5). 【建议】TODO注释以全部大写的字符串TODO开头，并带有写入括号内的姓名、email地址或其他可以标识负责人或者包含关于问题最佳描述的issue，随后是这里做什么的说明。

6). 【强制】不要在行尾添加分号或者用分号将两条命令放在同一行。

7). 【强制】不要在代码中使用太多的空行来区分不同的逻辑模块：

- 顶级函数和顶级类定义之间、导入语句与之后的业务代码之间保留两个空行。
- 使用1个空行来分隔类中的方法（method）定义。
- 任何文件最后应该保留一个空行。
- def行后无空行，在函数或方法内你认为合适的地方可以使用单空行。

8). 【强制】在二元运算符之前换行，对于有多个分句的语句避免将多行写在一起。

9). 【建议】单引号和双引号效果一样，但是不推荐将这两种方式看作一样并且混用，最好选择一种规则并坚持使用。如果字符串本身包含引号，可以混用。

10). 【强制】每行不超过120个字符，并适当使用换行符或者圆括号进行隐式行连接。

11). 【强制】不要在条件语句和返回语句中使用括号，除非值的类型是元组。

12). 【建议】在将来有可能扩展的列表末尾添加冗余的逗号。

13). 【强制】尾后逗号在定义单元素元组（tuple）时是必需的，为了清楚起见，使用括号（技术上来说是冗余的）括起来。

14). 【建议】条件表达式/三元运算符简单情况可以使用，每个部分（真值表达式、if表达式、else表达式）必须在一行内完成，简而言之条件表达式应该简单，不能太复杂。

15). 【建议】布尔值和True/False 做比较时，不建议用 `==`。

16). 【建议】使用f-string的方式格式化字符串，尽量避免使用`%`和`format`格式化。

17). 【建议】在执行主程序前总是检查 `if __name__ == '__main__'`。

18). 【**强制**】None比较使用`is`或`is not`，不使用`==`操作符。

19). 【**强制**】对象类型的比较始终使用`isinstance()`而不是直接比较。

20). 【建议】使用`is`比较对象的内存地址，使用`==`比较对象值。

21). 【建议】使用切片操作时，不建议无条件使用负步进值进行切片，会增加代码的理解难度。

#### 2、命名规范

1). 【建议】以单前导下划线开头的变量或方法仅供内部使用，不要外部访问。不建议使用双前导下划线，这会触发Name Mangling机制，使代码变得难以理解和维护。

2). 【**强制**】如果某个变量最合适的名称已被Python语言中的关键字占用（诸如`class`或`def`的名称不能用作Python中的变量名），在这种情况下，可以追加一个下划线来绕过命名冲突。

3). 【**强制**】命名时不要以双下划线开头和结尾，这种命名用来表示"magic"对象或用户控制命名空间中的属性（如`__init__`、`__import__`或`__file__`），不要擅自臆造，按照文档说明使用它们。

4). 【建议】模块命名应该短小，且为全小写。可适当使用下划线`_`，提高可读性。

5). 【**强制**】类命名应该简洁且具有描述性，遵循PascalCase的命名约定，尽量避免缩写，除非达成共识。

6). 【**强制**】函数命名必须都是小写，必要时使用下划线来提高可读性。只有当已有代码风格已经是混合大小写时（比如threading.py），为了保留向后兼容性才使用混合大小写。

7). 【建议】由于异常实际上也是类，因此类命名约定也适用于异常。不同的是，如果异常确实是一个错误的话，异常名应该加上"Error"后缀。

8). 【建议】常量命名使用大写字母，单词之间用 `_` 分割。

9). 【建议】不要使用字符`'l'`（L的小写字母）、`'O'`（o大写字母）或者`'I'`（i的大写字母）来作为单个字符的变量名。在一些字体中，这些字符和数字1和0无法区别开来。

#### 3、导入规范

1). 【建议】导入应该分行写，而不是都写在一行，但一个模块下的多个子类可以在一行导入。

2). 【建议】导入应该放在文件开头，位于模块（module）注释和文档字符串（docstring）之后，变量和函数声明之前。

3). 【建议】导入应该按照标准库imports、相关第三方包imports、本地包/模块的imports顺序分组来写，不同组的imports之前用空行隔开。

4). 【**强制**】所有导入的对象必须在当前文件中被使用，非必要不导入。

5). 【**强制**】避免使用通配符导入（形如`from <module> import *`），避免导入过多不必要的对象，也避免因模块变化导致导入内容发生改变，引起预期外的错误。

6). 【建议】绝对导入通常更易读，通常情况都应该使用绝对导入。

7). 【建议】优先导入package和module，尽量避免直接导入单个type或class引起命名冲突（使用`module.class`）。

#### 4、变量规范

1). 【建议】变量定义建议增加类型注解（type hints），尤其是变量结构比较复杂的情况，便于后续代码理解和使用数据结构。

2). 【建议】模块（包）级别的变量经初始化后不再修改，作为常量使用。如果程序依赖修改模块变量会产生线程间资源冲突。

#### 5、类编写规范

1). 【建议】property用来做接口封装和数据访问控制（只读变量）。类内被声明为property的接口只做数据访问（修改），不能有副作用。

2). 【建议】类定义中所有不依赖类成员变量（函数）的方法应该被声明为`@staticmethod`。

3). 【建议】类成员中如果包含不能直接调用的函数，这个类需要被定义为接口类，继承`abc.ABC`。

4). 【建议】ABC类内不能直接被调用的函数为接口函数，接口函数需要加`@abstractmethod`修饰。

#### 6、函数编写规范

1). 【**强制**】函数参数仅用作输入，不使用函数参数做结果返回。（类方法的self和cls参数不包含在本条"函数参数"范围）

2). 【**强制**】单个函数的行数不超过 80 行（不包含函数的docstring）。

3). 【建议】对外暴露的接口中不使用（`*args`、`**kwargs`）可变参数。

4). 【建议】基于面向对象组织代码逻辑。不定义全局函数，每个函数都应在类内部（除main函数），根据需要添加`@staticmethod`或`@classmethod`修饰。

5). 【建议】所有函数接口带类型注解（type hints）。

6). 【**强制**】仅使用常量作为函数参数的默认值，返回常量的函数本身不属于常量。

7). 【建议】有明确返回值的函数需要定义类型，在所有分支上显式 `return + 结果`（包括`return None`）。

8). 【建议】没有返回值的函数，通常不写return，在需要返回的分支上使用`return`（不加None）。

9). 【建议】return行应尽量简洁，避免复杂表达式。

#### 7、异常处理规范

1). 【建议】抛出的异常应该有明确类型（自定义派生类或built-in），不直接使用`Exception`异常。

2). 【建议】`assert`判断会隐式抛出`AssertionError`异常，但没有提供足够上下文信息，相当于try/raise的语法糖。AssertionError在整体工程中不便于区分处理，建议抛出明确的异常（不包含单测代码）。

3). 【建议】将try子句限制为必需的绝对最小代码量。

4). 【建议】捕获异常时，尽可能使用明确的类型，而不是用一个空的`except:`语句。空的except语句将会捕获到SystemExit和KeyboardInterrupt异常，很难区分程序的中断到底是Ctrl+C还是其他问题引起的。

5). 【建议】如果想对try块中的异常进行兜底处理，使用`except Exception:`（空`except:`等同于`except BaseException`）。

#### 8、语法糖规范

1). 【建议】避免在列表解析中使用复杂的逻辑或嵌套的条件语句。列表解析的目的是提供简洁的语法，因此过于复杂的逻辑可能会导致代码难以理解和维护。

2). 【建议】在处理大量数据或需要延迟计算的情况下使用生成器，可以按需生成值，而不是一次性生成所有值，从而节省内存提高性能。

3). 【建议】在使用lambda表达式时应遵守如下原则：

- 对于单行代码而言，可以使用lambda表达式，如果lambda表达式内的代码超过60-80个字符，最好定义成为常规的内嵌函数。
- 建议使用def表达式，而不是通过赋值语句将lambda表达式绑定到一个变量上。
- 不要调用会产生副作用的函数，如修改外部变量、写入文件、发送网络请求等，这样会导致不可预测性，调试困难等。

4). 【建议】filter和map的函数参数尽量不要使用lambda。建议使用列表解析和for循环，其他情况下可以使用filter和map。

5). 【建议】尽量避免使用反射。反射是一种强大而灵活的技术，但也容易导致代码复杂性和可读性的降低。因此，应该谨慎使用反射，并在确实需要动态访问对象属性和方法的情况下使用。

6). 【建议】字符串拼接`join`比`+`速度更快。使用`+`每次拼接都会创建一个新的字符串对象，导致额外的内存分配和拷贝操作。

7). 【建议】对于需要频繁创建大量线程的场景使用线程池，避免频繁的线程创建和销毁，同时通过线程池限制并发线程数量。对于I/O密集型任务优先考虑使用多线程；对于CPU密集型任务，优先考虑多进程，绕过GIL的限制。

#### 9、安全编程规范

1). 【**强制**】任何对外部系统的访问都需要设置超时参数，并对超时失败进行处理，避免调用外部系统长时间阻塞影响服务。

2). 【**强制**】任何对外部系统的访问地址避免写死节点列表，禁止在代码中硬编码ip、主机名，避免机器变更导致系统故障。优先使用域名、MGW VIP，或者使用注册发现等方式动态获取服务地址。

3). 【**强制**】Mysql查询语句不要使用事务。

4). 【建议】Mysql事务应保持简短，减少耗时的IO操作或网络请求，以避免对数据库的锁定时间过长，影响其他事务的执行。

5). 【建议】使用With语句显式释放资源。如果忘记关闭资源，可能会导致内存泄漏，资源没有正确关闭会导致资源泄漏，占用系统资源并最终耗尽内存。若资源未实现`contextlib.contextmanager`接口，也可通过try-finally，在finally语句中正确关闭资源，但注意不要在finally中使用return。

6). 【**强制**】明文密码不能硬编码到代码中。建议使用kms管理秘钥信息，秘钥通过kms平台管理，对可访问源做好授权，在使用秘钥时通过kms sdk读取。

7). 【**强制**】避免使用eval执行用户输入。用户输入的字符串可能包含恶意代码，应使用其他方法来解析和处理用户输入，例如使用`ast.literal_eval()`来安全地评估字符串表示的字面值。

8). 【**强制**】禁止使用空口令、弱口令。口令强度须同时满足：密码长度大于14位；必须包含大小写英文字母、数字、特殊字符；不得使用各系统、程序的默认初始密码；密码要经常变更，且禁止传递明文口令。

9). 【**强制**】按输入的数据类型进行数据校验，校验内容包括但不限于：数据长度、数据范围、数据类型与格式。如果检测到参数不合法，不要尝试修正，而应立即拒绝。

10). 【**强制**】当程序需要从用户指定的URL地址获取网页文本内容、加载指定地址的图片、进行下载等操作时，需要对URL地址进行安全校验，避免SSRF漏洞。

11). 【**强制**】使用XML传输数据时，禁用XML外部实体，预防XXE攻击。

12). 【**强制**】禁止使用不安全的反序列化函数。禁止使用`yaml.unsafe_load()`函数反序列化YAML数据，建议使用`yaml.safe_load()`代替。

13). 【**强制**】设置正确的HTTP响应包类型。响应包的HTTP头`Content-Type`必须正确配置响应包的类型，禁止将非HTML类型的响应包设置为`text/html`，否则攻击者可以在响应包中插入恶意的HTML代码，极易引发XSS漏洞。如果响应包的类型`Content-Type`需要为`text/html`时，必须对响应体进行编码处理。

14). 【**强制**】使用参数化查询／预编译，强制区分数据和命令，避免产生SQL注入漏洞。推荐使用ORM框架如SQLAlchemy或Django ORM，可以自动处理参数化查询。

15). 【**强制**】禁止使用`eval`、`exec`、`execfile`等函数处理可以外部输入的数据。这些函数可以执行系统命令及Python代码，在输入的内容不确定时，黑客可以通过植入代码的方式控制进程甚至是服务器。

16). 【**建议**】文件读取避免接受外部参数拼接。保存文件目录建议后端写死，并对文件名进行校验（字符、字符类型、长度），避免外部可控。

17). 【**强制**】避免使用不安全的加密算法。

| 算法 | 禁止使用 | 推荐使用 | 兼容使用 | 场景 |
| --- | --- | --- | --- | --- |
| 对称加密算法 | DES、RC4、Blowfish、Twofish | AES-256 | AES-128 | 生产场景高并发大规模数据加解密 |
| 完整性算法 | MD5 | SHA-256 以上的SHA-2系列 | SHA-1（只在HMAC和KDF中使用，禁止在TLS证书或者随机数生成器中使用） | 用于在数字签名中提供完整性功能 |
| 验证算法 | HMAC-MD5 | HMAC-SHA-256、SHA-2、HMAC-SHA-384、HMAC-SM3（国密） | HMAC-SHA1 | 常见用来做BA验证 |
| 密码哈希算法 | MD5 | SHA-512 | SHA-256 | 不可逆方式存储用户凭据如密码 |

18). 【**强制**】系统进行鉴权，推荐接入公司鉴权系统。在接入认证前提下，无论内外部系统，均需要接入公司鉴权系统如IAM、UAC/UPM，并遵循最小权限原则，对访问系统人员的权限进行控制。

19). 【**强制**】避免对外暴露程序异常信息。应合理使用try/except/finally等捕获系统异常，避免错误信息输出到前端而被外部人员研究分析。同理，对外环境禁止开启debug模式，或将程序运行日志输出到前端。

20). 【**强制**】重要行为都记录日志。重要行为（时间、IP、username、url、内容、成功失败等）都需进行日志记录，且通过异地备份等方式可靠保存6个月以上，管理员禁止删除日志，日志入口需要鉴权。

21). 【**强制**】禁止在异常、日志中保存敏感信息。不能在异常、日志中保存密码（包括明文密码和密文密码）、密钥、CVV、有效期和其它C3C4级别敏感信息。

22). 【**强制**】生产环境必须关闭flask Debug模式。

23). 【**强制**】生产环境必须关闭Django Debug模式。

#### 10、常用内建库使用规范

1). 【**强制**】使用subprocess模块执行Linux命令，不使用`os.system`、`os.popen`、`commands`等模块。subprocess会fork子进程，故需要正确回收子进程的 pid，通过调用`Popen.communicate`或者`Popen.wait`实现，简单场景可以直接使用`subprocess.run`方法。

2). 【建议】在处理不信任的数据时，禁用pickle进行序列化和反序列化。pickle在序列化时可以执行Python代码，因此存在远程代码执行和代码注入的风险，在处理不信任数据时（比如外部接口的返回），建议使用json模块。

#### 11、日志规范

1). 【建议】按公司规范要求，所有日志应该写到`/opt/logs`下，不能把日志写到`/`根目录下，根目录大小有限且受镜像挂盘限制，而`/opt`目录是可扩展的。

2). 【**强制**】任何程序都应该控制日志文件大小，避免打满磁盘存储空间，必要时对日志进行滚动切割。

#### 12、静态类型检查规范

1). 【建议】python3项目开启静态类型检查，可以避免一些常见的运行时类型错误，比如NPE等。

2). 【建议】建议开启strict检查模式，可以避免绝大多数运行时的类型错误。

3). 【建议】每个module使用 `# type: ignore` 跳过的检查最好不要超过3个。

4). 【建议】非私有函数的入参和返回，使用类型提示。

5). 【建议】对协程（async/await）使用类型提示。

6). 【建议】变量如果可以为None，使用`Optional`类型提示。

7). 【建议】对模糊类型的类型提示，优先使用`object`，而不是`Any`。如果类型为`Any`，相当于跳过了类型检查；如果类型标注为`object`，依然会继续进行类型检查，在具体使用的地方，可以使用`isinstance`进行类型判断，避免`AttributeError`。

### 四、单元测试

1). 【强制】为了保证单元测试稳定可靠且便于维护，单元测试用例之间不能互相调用，也不能依赖执行的先后次序。

2). 【强制】单元测试通常在PR过程频繁触发执行，执行过程完全自动化才高效，需要人工参与修改配置或者验证结果的单元测试不是一个好的实践。单元测试中不准使用 print 来进行人肉验证，必须使用 assert 来验证。

3). 【强制】单元测试通常会被放到持续集成中，每次有代码PR单元测试都会被执行。如果单测对外部环境（网络、数据库、业务服务等）有依赖，容易导致持续集成机制的不可用。

4). 【强制】单测粒度至多是类级别，一般是方法级别，只有测试粒度小才能在出错时尽快定位到出错位置。单测不负责检查跨类或者跨系统的交互逻辑，那是集成测试的领域。

5). 【强制】单测不允许使用恒真断言，不允许使用弱测试断言。

6). 【强制】测试程序应该能运行多次保持结果一致性，保证测试是幂等性。

7). 【强制】不要有流程控制语句，一个好的测试用例处理顺序简单直接，准备数据，验证结果，如有必要，使用子方法分解结构，让测试用例更加易读。

8). 【强制】Bug修复、新增功能时，应该增加新的单测用例。

9). 【强制】单元测试代码写在如下工程目录：`test/test_模块xx`，不允许写在业务代码目录下；测试类以 **Test** 开头，测试用例以 **test_** 开头。

10). 【建议】编写单元测试代码遵守 BCDE 原则，以保证被测试模块的交付质量：B（Border）边界值测试，包括循环边界、特殊取值、特殊时间点、数据顺序等；C（Correct）正确的输入，并得到预期的结果；D（Design）与设计文档相结合来编写单元测试；E（Error）强制错误信息输入（如非法数据、异常流程、非业务允许输入等），并得到预期的结果。

11). 【建议】测试用例的变量和方法都要有明确的含义。

12). 【建议】测试用例易读，测试用例代码应该遵循规范，像应用代码一样易于理解。

13). 【建议】不要在自己的类中测试第三方的类库，类库应该由它们自己的测试用例来测试。如果它们自己没有，应该考虑使用mock来模拟类库的输出结果，确保输入数据的确定性。

14). 【建议】对于不可测的代码建议做必要的重构，使代码变得可测，避免为了达到测试要求而书写不规范测试代码。

15). 【建议】单元测试作为一种质量保障手段，不建议项目发布后补充单元测试用例，建议在项目提测前完成单元测试。
