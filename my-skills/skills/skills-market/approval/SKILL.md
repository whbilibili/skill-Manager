## 一、模糊意图追问规则

收到模糊指令时，**先追问，不猜测，不执行**。
| 用户说 | 追问 |
|---|---|
| 「看一下我的审批」「我的审批」 | 是要看：待审批 / 我发起的 / 已审批 / 抄送我的？ |
| 「帮我同意/通过一下」 | 请提供单据名称或编号，或先查询待审批列表 |
| 「最近的审批」 | 是发起的还是待处理的？时间范围是？ |
| 「查一下某某人的审批」 | 是查他发起的，还是他审批过的？ |
| 「由我审批的」「我来审的」「我审的」「我审批过的」 | 是查**待审批**（还没处理的），还是**已审批**（历史记录）？ |
| 「那个单据」「刚才那条」 | 如有多条同类单据，展示列表让用户选编号 |
| 「这个单子审批到哪了」「现在在谁手里」（未说明角色） | 您是该单据的**发起人**、**审批人**，还是**抄送人**？（决定用哪个列表接口反查 billId） |
| 「全部通过」「全部批掉」「一键审批」 | **必须追问范围**：是当前所有待审批，还是某类流程？请说明关键词或条件后再执行 |
| 「把XXX都通过」「XXX全批了」（无筛选条件） | **必须追问范围**：是当前所有待审批，还是某类流程？请说明关键词或条件后再执行 |

## 零、缺失参数反查原则（通用）

所有操作所需的 `taskId` / `billId`，若上下文不存在，**禁止让用户手动提供，禁止直接调用操作接口**，必须先从对应列表接口反查：
| 所需参数 | 反查接口 | 适用场景 |
|---|---|---|
| `taskId` | `getPendingApprovals` | 加签操作（仅限当前待您审批的单据）、转交操作 |
| `billId`（待审批） | `getPendingApprovals` | 审批流查询（审批人 · 待处理） |
| `billId`（已审批历史） | `getHandledApprovals` | 审批流查询（审批人 · 历史已处理） |
| `billId`（已发起） | `getSubmittedApprovals` | 审批流查询（发起人）、撤回操作 |
| `billId`（抄送） | `getCCApprovals` | 审批流查询（抄送人，仅可查进度，无法操作） |

> 若角色不明确，先追问「您是发起人、审批人，还是抄送人？」再按对应路径查询。

## 二、单条审批操作 SOP

> 触发：「通过」「驳回」「同意」+ 指定单据。**严格按步骤执行，不得跳步。**
> **Step 1 · 确定目标单据**

- 若未指定，调用 `getPendingApprovals` 展示列表，等用户选择
- 自然语言指定（「那个设备借用」）→ 若多条匹配则展示列表让用户选编号
- 出口：明确了单条目标单据
  **Step 2 · 校验可操作性**
- 检查 `directOperate` 字段
- `支持` → 进入 Step 2.5
- `不支持` → 告知「该单据未开启快捷审批，请[点击此处]({detailUrl})前往浏览器操作」，流程结束
- 抄送人身份（API 返回数据中无操作权限标识）→ 告知「您是该单据的抄送人，无审批权限」，流程结束；若无法判断身份，保守处理：拒绝操作并提示用户确认角色
  **Step 2.5 · 业务路由检查**
- 取当前单据的 `processId`，查 `references/process-routing.md`
- 命中路由表 → 执行路由逻辑（详见十三章），缓存风险信息，在 Step 3 二次确认时一并展示
- 未命中路由表 → 跳过，直接进入 Step 3
  **Step 3 · 二次确认**
- 展示：流程名称（附 `detailUrl` 链接）、发起人、摘要信息、操作类型（通过/驳回）
- 示例格式：「即将**通过** [设备借用申请](detailUrl)，发起人：张三，摘要：XXX。确认吗？」
- 等待用户明确回复「确认」「是」「好」等肯定词
- 未收到明确确认 → 不执行，保持等待
  **Step 4 · 执行并验证**
- 调用 `operateApprove`，注意 `--operationType` 与 `--operationUrl` 必须配对（APPROVE↔approveUrl / REJECT↔rejectUrl），参见 `references/cli-cheatsheet.md`
- 若用户提供了审批意见则附加 `--message "<意见内容>"`
- 返回成功 → 展示：`✅ 审批完成 | [流程名称](detailUrl) | 发起人：XXX | 操作：通过/驳回`
- 返回失败 → 参照「错误处理」说明原因

## 三、批量审批

> 触发：「批量通过」「帮我把 XXX 批掉」「全部审批」「自动审批」「以后收到 XXX 自动帮我通过」
> 详细流程见 → [references/workflow-batch.md](references/workflow-batch.md)
> **入口规则：** 0. **判断单条 vs 批量**：凡用户指令涉及条件筛选（如「张三发起的」「B级的」「所有XXX」「都通过」），无论触发词形式，均走批量 SOP；不得拆解为单条 SOP 多次执行。

1. AI 从用户指令提取筛选规则（关键词 / 发起人 / 时间范围）；**若无任何筛选条件（如只说「全部通过」），必须先追问范围，不得直接拉取全量列表**
2. 调用 `getPendingApprovals` 获取候选列表
3. 筛选 `directOperate: 支持` 的单据，标记跳过项（三方流程）
   3.5. **批量路由扫描**：遍历候选列表，识别每条单据的 `processId`；命中路由表的单据 → 调用对应业务 Skill 取风险信息（详见十三章），随清单一并展示；调用失败 → 展示「⚠️ 风险信息不可用」，不阻断清单展示
4. **向用户展示清单（每条附 `detailUrl` 链接），等待「确认」后才执行**
5. 逐条串行执行（不并发），**单次上限 50 条**；超出 50 条时展示前 50 条清单并提示「共发现 N 条，本次最多执行 50 条，是否继续？如需处理全部请分批操作」
6. **单条执行失败时，标记失败并继续处理其余条目，不中断批量流程**
7. 完成后发大象通知
   **限流声明：** 批量操作调用 API 间隔不少于 200ms（串行执行已保证），避免对审批系统造成频繁请求压力。
   **出口：** 所有 `directOperate: 支持` 的单据处理完毕，操作结果汇总已展示，跳过项已列明原因

---

**规则型自动审批（进阶：持续生效）：**
用于「以后每次」自动执行的场景，例如：

- 「以后 C2/C3 密级数据权限审批，收到后自动帮我通过」
- 「张三发起的出差申请都直接批」
  **Step A · 自然语言 → 结构化规则（每次必须执行）**
  收到规则描述后，**必须实时调用 `getPendingApprovals` 获取最新待审批列表**，再按规则筛选并向用户展示匹配结果确认（不得使用会话上下文中的旧数据）：

```
我理解的规则如下，请确认：
- 流程关键词：数据权限申请
- 数据密级：C2 或 C3
- 操作：自动通过
- 备注：不符合以上条件的单据不处理
确认后我将按此规则执行。如需修改请直接说。
```

用户确认后，引导创建定时任务，并提供频率选项供快速选择：

```
可以帮你创建定时任务，请选择执行频率：
1. 每天早上 9 点
2. 每天下班前 6 点
3. 每周一早上 10 点
4. 每半小时（最高频率）
5. 自定义（请描述）
注意：最短间隔为每 30 分钟一次。
```

- 用户选择后，使用 daxiang-scheduled-message skill 创建 cron 任务
- **最短执行间隔：30 分钟**，低于此频率的请求须拒绝并说明原因
- **不得在用户不知情的情况下后台执行**

## 四、加签操作

> 触发：「加签」「前加签」「后加签」「加一个审批人」「Append」

- 必须参数：**taskId + 加签类型（前加签/后加签）+ 被加签人 + 加签意见**
- **用户未说明前/后加签类型时，必须先追问，不得猜测**
- 加签意见、被加签人必填，若用户未提供，需追问
  **taskId 获取路径（⚠️ 重要）：**
  参见「零、缺失参数反查原则」。`taskId` 仅能从 `getPendingApprovals` 中获取，**仅限当前待您审批的单据**；若列表中未找到，告知「未在待审批列表中找到该单据，加签仅限当前待您审批且尚未处理的单据」，流程结束。
- 用户输入的可能是姓名或 MIS 号，**必须调用 `oa-skills shenpi queryUserInfo --filter <mis/姓名>` 查询，取返回结果的 userId 字段作为 --targetUserId 值**：
  - 若查询结果为多条，展示列表让用户确认
  - 若查询结果为空，告知「未找到该用户，请确认姓名/MIS 是否正确」
- 参数无误后，**执行前必须展示二次确认**，格式如下：
  ```
  您将执行【后加签/前加签】操作：
  - 单据：<billId>
  - 被加签人：<姓名>（<userId>）
  - 加签意见：<意见>
  确认执行吗？
  ```
  用户确认后调用 `addSignApprove` 执行加签操作
- 成功后回复「✅ 加签成功」；失败则参照错误处理

## 五、催办

> 触发：「催一下」「催办」「提醒审批人」「催催审批」「催一批」「把 XXX 催一下」

### 5.1 单条 vs 批量判断

1. 调用 `getSubmittedApprovals --billStatus 1` 获取审批中的单据
2. 结果为 **1 条** → 走「单条 SOP」
3. 结果为 **多条** → 若用户已明确指定单条目标则走单条；否则展示清单走「批量 SOP」
4. 结果为 **0 条** → 回复「当前没有审批中的单据」，流程结束

> 若用户说「全部催」但未指定范围 → 追问：「是要催办所有审批中的单据吗？请确认范围。」

---

### 5.2 单条催办 SOP

**Step 1 · 确认目标单据**

- 若上下文已明确单条目标，直接进入 Step 2
- 若未明确，调用 `getSubmittedApprovals --billStatus 1`，多条时展示列表让用户选择

**Step 2 · 二次确认**

```
即将催办 [流程名称](detailUrl)，摘要：XXX，发起于 XXXX/XX/XX。
（每个单据每天只允许催办一次）确认吗？
```

- 等待用户明确回复「确认」「是」「好」等肯定词，未收到则不执行

**Step 3 · 执行并响应**

取得 `platformId` 及必须参数（见下表）后调用 `urgeApproval`：

| 平台     | platformId | 必须参数       | 参数来源                                    |
| -------- | ---------- | -------------- | ------------------------------------------- |
| 审批平台 | `1`        | `--billId`     | 已发起列表的 `billId` 字段                  |
| 快搭平台 | `14`       | `--procInstId` | 已发起列表的 `billId` 字段（即 procInstId） |

> ⚠️ 快搭平台（platformId=14）用 `--procInstId` 传参，**不是** `--billId`；两者参数名不同，不得混用。

接口返回处理：

- 返回含 `message` 字段且不为空 → 直接展示 message 内容（如「今天已经催办过，请明天再试」）
- 返回 `null` 或无 message → 回复「✅ 催办成功，已提醒审批人」
- 接口报错 → 参照错误处理

---

### 5.3 批量催办 SOP

**Phase 1 · 提取筛选条件**
从用户指令提取（可选）：流程名关键词 / 时间范围；未指定则催办所有审批中单据（需先确认范围）

**Phase 2 · 获取候选列表**

```bash
oa-skills shenpi getSubmittedApprovals --billStatus 1
```

- 超过 50 条 → 提示「共 N 条，单次上限 50 条，建议缩小范围或分批操作」

**Phase 3 · 二次确认清单（必须执行，不可跳过）**

```
即将催办以下 N 条单据（每个单据每天只允许催办一次）：
| # | 流程名称 | 摘要 | 发起时间 |
|---|---|---|---|
| 1 | [采购申请](url) | 打印纸 CNY79 | 2026/03/30 |
| 2 | [采购申请](url) | 签字笔 CNY45 | 2026/03/30 |
输入「确认」开始执行，或「取消」中止。
```

- 未收到明确确认 → 不执行

**Phase 4 · 逐条串行执行**

- 串行执行，不并发；每条间隔 300ms
- 单条失败 → 记录错误，继续下一条，不中断整体流程

**Phase 5 · 汇总结果展示**

- **全部成功（无异常）**：`✅ N 条单据催办完成`（一行，不展开明细）
- **有异常（部分已催过或失败）**：展开明细：

```
催办完成：✅ 成功 N 条  ⚠️ 今日已催 N 条  ❌ 失败 N 条
| # | 流程名称 | 结果 | 说明 |
|---|---|---|---|
| 1 | [采购申请](url) | ✅ 成功 | - |
| 2 | [采购申请](url) | ⚠️ 今日已催 | 今天已经催办过，请明天再试 |
| 3 | [采购申请](url) | ❌ 失败 | 接口返回错误：XXX |
```

**边界**：列表为空→「当前没有审批中的单据」；超50条→提示缩小范围或分批；执行中断→记录已完成条目并汇报。

## 八、转交审批

> 触发：「转交」「转给别人」「转交审批」「转移」「让 XX 处理」
> ⚠️ **仅支持快搭平台（platformId=14）**；审批平台（platformId=1）不支持，告知用户后流程结束

**Step 1 · 收集信息（可并行）**

以下三项信息若用户未一次性提供，逐项追问，**不得跳步执行**：
1. **审批单据链接**：从 `https://kuaida.sankuai.com/app-xxx/workflow?procInstId=<id>&taskInstId=<id>` 提取 `procInstId`；链接中的 `taskInstId`（如果有）禁止直接使用，**必须通过下一步反查真实 taskId**
2. **转交目标用户**：用户提供姓名或 MIS 号 → 调用 `queryUserInfo --filter <姓名/MIS>` 查询 `userId`；多条结果展示列表让用户确认；转交目标不能是操作人自己
3. **转交意见**（选填）：主动询问「转交意见是什么？」，用户回复「无」则跳过不传 `--comment`

**Step 2 · 反查真实 taskId**

调用 `getApproveFlowDetail --billId <procInstId> --platformId 14`，筛选**待审批（进行中）**的审批任务节点，提取 `taskId` 作为 `taskInstId` 入参：

- 若只有**1 个**待审批任务 → 直接使用该 `taskId`
- 若有**多个**待审批任务 → 展示列表让用户选择：
  ```
  当前有多个待审批任务，请选择要转交的任务：
  1. 审批人：张三 | 节点：XXX审批
  2. 审批人：李四 | 节点：XXX审批
  ```
- 若**没有**待审批任务 → 告知「该单据当前无待审批任务，无法转交」，流程结束

**Step 3 · 二次确认并执行**

```

- 单据：<procInstId>
- 转交给：<姓名>（<userId>）
```

- 用户确认后调用 `oa-skills shenpi transferApproval --procInstId <procInstId> --taskInstId <taskId> --targetUserId <userId> --comment <转交意见> # 转交意见可选` 执行转交
- 转交成功后（接口返回null）回复「✅ 转交成功，已转交给 <姓名> 审批」；失败则参照错误处理

## 十、审批流详情

> 触发：「查询审批进度」「审批流详情」「当前在谁手里」「审批到哪了」

- 调用 `getApproveFlowDetail --billId <id> --platformId <1|14>` 获取审批流详情
- 获取数据后，**读取 `references/flow.md`**，按其规则渲染展示结果
- platformId：1=审批中心，14=快搭；若接口返回 500 报错，尝试切换另一个 platformId 重试一次
  **billId 获取路径（⚠️ 重要）：**
  参见「零、缺失参数反查原则」，按角色身份选择对应反查接口：
- **审批人（待处理）**（「那个单子审批到哪了」）→ `getPendingApprovals`
- **审批人（历史已处理）**（「我之前批过的那个单子现在状态怎样」）→ `getHandledApprovals`
- **发起人**（「我发起的那个单子现在到谁手里了」）→ `getSubmittedApprovals --billStatus 1`（默认查进行中，减少干扰项）
- **抄送人**（「我抄送到的那个采购单进度如何」）→ `getCCApprovals`；注意：抄送人仅可查进度，无法执行审批操作
- 若角色不明确 → 追问「您是该单据的发起人、审批人，还是抄送人？」
- 禁止在 billId 未知的情况下直接调用 `getApproveFlowDetail`
- 用户直接提供 billId 时：先尝试 `--platformId 1`，返回 500 则自动切换 `--platformId 14` 重试，无需追问用户

## 十二、查询结果展示规范

- 「已审批」和「抄送我的」**分离为两个独立表格，不合并**
- 默认展示最新 **10 条**，末尾提示「如需更多请说明」
- 摘要完整保留，超 3 行用 `<details>` 折叠
- 停留时长人性化显示：< 1h 显示分钟（5m）、< 24h 显示小时（3h）、< 30d 显示天（7d）、≥ 30d 显示「30d+」；计算方式：当前时间 - arrivalTime
  > ⚠️ **时间计算必须执行代码**（`python3 -c` 或等效方式），严禁心算估算。时间戳单位为毫秒，除以 1000 得到秒，再换算。
  > ⚠️ **严禁将内部字段名输出给用户**（如 `directOperate`、`billId`、`taskId`、`platformId`、`tenantId` 等）。这些字段仅用于内部逻辑判断，不得以任何形式展示。遇到需要向用户解释原因时，使用用户能理解的白话描述。例如：不说「directOperate 为不支持」，而说「该单据不支持快捷审批，请前往浏览器操作」。

### 表格列定义

| 分类     | 列定义                                                                           |
| -------- | -------------------------------------------------------------------------------- |
| 待审批   | 流程名称、发起人、摘要信息、提交时间、停留（按到达时间倒序，即最新收到的排最前） |
| 已发起   | 流程名称、摘要信息、发起时间、审批进度                                           |
| 已审批   | 流程名称、发起人、摘要信息、我的操作、处理时间                                   |
| 抄送我的 | 流程名称、发起人、摘要信息、流程状态、抄送时间                                   |

**待审批列表路由标注规则：**
展示待审批列表时，扫描每条单据的 `processId` 是否命中 `references/process-routing.md` 路由表：

- 命中 → 流程名称后追加 ` 📊 可查风险`（如「采购门户\_采购预算申请单审批 📊 可查风险」），明确告知用户该单据有业务风险分析可供参考
- 未命中 → 正常展示，不追加标记
- 列表底部追加提示行：`📊 = 可查风险，回复「查看第 X 条风险」获取详情`
- 此步骤仅做标注，不调用业务 Skill，不增加列表加载耗时

审批进度格式：`审批中` / `审批完成` / `已驳回` / `已撤回`
结果枚举：通过 / 驳回

### 操作结果格式

汇总：`✅ 已通过 N 条  ❌ 已驳回 N 条  🔴 执行失败 N 条`
表格列：编号 / 流程名称（附 `detailUrl` 链接） / 发起人 / 结果 / 说明
排序：✅ → ❌ → 🔴

## 十五、业务 Skill 路由

> 适用场景：当待审批单据的流程编码命中路由表时，自动调用对应业务 Skill 补充风险信息，展示给审批人作为决策参考。
> **路由触发规则（三种时机）：**
> ① **列表展示时（标注）**：`getPendingApprovals` 返回列表后，扫描每条单据的 `processId`，命中路由表的单据在流程名称后追加 `📊 可查风险` 标记（仅标注，不自动调用业务 Skill），并在列表底部追加提示行引导用户操作。用户主动要求查看某条风险详情时（如「看一下第 X 条的风险」「查看详情」），再触发 Step 1-4 调用。
> ② **单条审批操作前**：二章 Step 2.5 主动调用。
> ③ **批量审批操作前**：三章第 3.5 步主动调用。
> 识别字段：单据的 `processId`（来自 `getPendingApprovals` 返回值，Agent 在上下文中缓存，无需再次请求）
> 路由表：见 [references/process-routing.md](references/process-routing.md)
> **执行流程：**

```
Step 1：识别 processId 是否命中路由表
  来源优先级：
  - 若已通过 getPendingApprovals 列表获取过该单据，直接使用上下文中缓存的 processId
  - 若未经列表（用户直接提供 billId），调用 getPendingApprovals --billId <billId> 尝试获取 processId
    → 若列表返回该单据：取 processId 继续
    → 若列表中未找到（非当前待审批单据）：静默跳过路由，不触发业务 Skill，正常展示审批信息
  命中路由表 → Step 2
  未命中路由表 → 正常展示，不触发路由
Step 2：检测业务 Skill 是否已安装
  检测方法：检查路由表中该条目对应的业务 Skill 目录是否存在于 ~/.openclaw/skills/
  已安装 → Step 3（直接调用）
  未安装 → Step 2a
Step 2a：向用户说明价值，询问是否安装
  从路由表（references/process-routing.md）读取该条目的「业务域」和「Skill 名称」
  展示：「检测到当前审批单涉及{业务域}流程。安装「{Skill名称}」后，我可以在审批时自动为你提供该流程的风险分析和辅助决策信息。是否现在安装？」
  用户确认 → Step 2b
  用户拒绝 → 展示「⚠️ 业务风险信息不可用」，不阻断，继续展示审批信息
Step 2b：从路由表读取 Skill ID，执行安装
  从 references/process-routing.md 读取该条目的 skillId 字段
  执行：mtskills i mt --id <skillId> -g
  安装成功 → 提示「✅ 安装成功，正在获取风险信息...」→ Step 3
  安装失败 → 展示「⚠️ 安装失败，风险信息暂不可用，你可稍后手动安装」，不阻断
Step 3：调用对应业务 Skill，传入 billId
  调用成功 → Step 4
  调用失败 / 无响应 → 展示「⚠️ {业务域}风险信息不可用」，继续展示审批流程信息，不阻断
Step 4：格式化风险结论
  有明确风险等级（高/中/低）→ 展示：风险等级 + 摘要 + 核心字段
  无明确风险等级 → 展示：摘要 + 核心字段（不展示风险等级行）
  ⚠️ 风险等级必须来自业务 Skill 原文明确表述，严禁 AI 自行推断
```

**展示格式示例（有风险等级）：**

```
📊 {业务域}风险信息（由{Skill名称}提供）
风险等级：高
摘要：{业务 Skill 返回的摘要}
关键数据：{业务 Skill 返回的核心字段}
建议：{业务 Skill 返回的建议}
```

**展示格式示例（无明确风险等级）：**

```
📊 {业务域}风险信息（由{Skill名称}提供）
摘要：{业务 Skill 返回的摘要}
关键数据：{业务 Skill 返回的核心字段}
```

**展示格式示例（调用失败）：**

```
⚠️ {业务域}风险信息不可用
```

**重要原则：**

- AI 不得基于审批中心已有信息自行推断业务风险等级，只透传业务 Skill 的结论
- 风险信息仅供参考，审批人最终决策，AI 建议不替代人工判断
- 业务 Skill 调用失败时，不影响审批操作的正常进行
