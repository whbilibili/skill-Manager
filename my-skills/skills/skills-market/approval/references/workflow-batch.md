## 触发词

「批量通过」「帮我把 XXX 批掉」「全部审批」「自动审批」「一键通过」「以后收到 XXX 自动帮我通过」「张三发起的 XXX 都直接批」

## 完整执行流程

### Phase 1 · 提取筛选规则

从用户指令中 AI 提取以下维度（可选，未提及的不限制）：
| 维度 | 示例 |
|---|---|
| 流程名关键词 | 「出差申请」「设备借用」 |
| 发起人 | 「张三发起的」 |
| 时间范围 | 「今天的」「本周的」「最近 3 天」 |
| 操作类型 | 通过（默认）/ 驳回 |
若用户说「全部通过」但未指定范围 → 追问：「是要通过所有待审批单据吗？请确认范围。」

### Phase 2 · 获取候选列表

```bash
oa-skills shenpi getPendingApprovals --limit 50 --offset 1
# 如有时间范围：
oa-skills shenpi getPendingApprovals --starttime '2025-03-03 09:00' --endtime '2025-03-31 23:30:30' --limit 50 --offset 1
# 如有发起人（需先获取其 userId）：
oa-skills shenpi getPendingApprovals --sponsorId {userId} --limit 50 --offset 1
```

- 总数超 50 条 → 提示「共 N 条，本次最多处理 50 条，建议缩小范围或分批操作」
- 查询失败 → 「审批系统暂时无法访问，请稍后重试」，流程终止

### Phase 3 · 筛选与标记

对获取到的列表进行过滤：
| 条件 | 处理 |
|---|---|
| 关键词匹配（流程名含目标词） | ✅ 纳入候选 |
| `directOperate: "支持"` | ✅ 可执行 |
| `directOperate: "不支持"` | ⚠️ 标记「跳过（三方流程）」 |
| 不匹配关键词 | — 不展示 |

### Phase 4 · 二次确认（必须执行，不可跳过）

向用户展示清单表格：

```
即将执行批量审批，请确认：
| # | 流程名称 | 发起人 | 摘要 | 操作 |
|---|---|---|---|---|
| 1 | 出差申请 | 张三 | 北京 2 天... | ✅ 通过 |
| 2 | 设备借用 | 李四 | MacBook... | ✅ 通过 |
| 3 | 合同审批 | 王五 | ... | ⚠️ 跳过（三方流程） |
共 2 条将执行，1 条跳过。
输入「确认」开始执行，或「取消」中止。
```

- 未收到明确确认 → 不执行
- 用户说「取消」→ 回复「已取消，未执行任何操作」

### Phase 5 · 逐条串行执行

```
规则：
- 串行执行，不并发
- 单条失败 → 记录错误，继续下一条（不中止）
- 每条执行间隔 300ms（避免频控）
```

调用示例：

```bash
# operationUrl 取值：operationType=APPROVE 时用 approveUrl，REJECT 时用 rejectUrl（来自单据详情字段）
oa-skills shenpi operateApprove \
  --billId "{billId}" \
  --operationType "APPROVE" \
  --operationUrl "{approveUrl}" \
  --platformId {platformId} \
  --taskId "{taskId}" \
  --tenantId "{tenantId}"
```

### Phase 6 · 汇总结果展示

执行完毕后在对话中展示：

```
批量审批完成
✅ 已通过 N 条  ❌ 已驳回 N 条  🔴 执行失败 N 条  ⚠️ 跳过 N 条
| # | 流程名称 | 发起人 | 结果 | 说明 |
|---|---|---|---|---|
| 1 | 出差申请 | 张三 | ✅ 通过 | - |
| 2 | 设备借用 | 李四 | ✅ 通过 | - |
| 3 | 合同审批 | 王五 | 🔴 失败 | API 返回错误：单据已处理 |
排序：✅ → ❌ → 🔴 → ⚠️
```

### Phase 7 · 大象通知

执行完成后，通过大象发送通知（daxiang-channel skill）：
**通知内容模板：**

```
【批量审批完成】
✅ 通过 {success} 条
🔴 失败 {fail} 条
⚠️ 跳过 {skip} 条（三方流程）
{如有失败}失败单据：
- {流程名称}（{发起人}）：{失败原因}
操作时间：{YYYY-MM-DD HH:mm}
```

## 边界情况处理

| 情况             | 处理                                                                 |
| ---------------- | -------------------------------------------------------------------- |
| 候选列表为空     | 「未找到符合条件的待审批单据」                                       |
| 全部为跳过项     | 「所有匹配单据均为三方流程，无法快捷操作，请[前往审批系统]手动处理」 |
| 超过 50 条       | 「共 N 条符合条件，单次上限 50 条，建议缩小范围或分批操作」          |
| 用户未说操作类型 | 默认「通过」，在确认清单中明确标注                                   |
| 执行过程中断     | 记录已完成/失败条目，最终汇报实际状态                                |
| 部分单据已处理   | 跳过并在结果中标注「单据已处理」                                     |

---

## 失败处理规范

单条 `operateApprove` 返回非 `true` 时：

1. 记录错误信息（错误码 + 描述）
2. 继续执行下一条（不中止整体流程）
3. 最终结果表格中标注 🔴 + 说明原因
4. 大象通知中列出失败单据
